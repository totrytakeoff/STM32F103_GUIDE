# AI 辅助 STM32 项目路线二行动手册

适用目标:

- AI 主写大部分代码
- 自己负责读懂、测试、修改、复盘
- 做出稳定项目，同时真正提升 STM32 能力

核心原则:

> AI 可以写 80% 的代码，但你必须掌握 20% 的关键点。

这 20% 包括: 接线、外设选择、数据流、初始化位置、启动函数、错误现象、验证方法。

## 1. 路线二是什么

路线二不是完全不写代码，也不是所有代码都自己憋。

它的工作方式是:

```text
你提出需求
-> AI 拆方案
-> AI 写最小代码
-> 你烧录测试
-> 你反馈现象
-> AI 帮你定位
-> 你读懂关键逻辑
-> 你做一个小修改
-> 你记录经验
```

你不需要背大量 HAL 函数，也不需要记住所有寄存器。你需要练的是:

- 看得懂代码在干什么
- 知道硬件信号怎么走
- 能根据现象判断大概是哪类问题
- 能改参数、改逻辑、改接线说明
- 能用测试证明功能是真的稳定

## 2. 每个功能的标准开发流程

每次只做一个小功能。不要一次把 LCD、串口、RTOS、按键、传感器全塞进去。

### Step 1: 自己写一句需求

示例:

```text
STM32F103 使用 USART1 接收 PC 发来的 4 行文本，并显示到 12864 LCD。
```

### Step 2: 让 AI 拆方案

可以这样问:

```text
这个功能需要哪些外设？数据流是什么？可能有哪些硬件和软件坑？
先不要写代码，只帮我拆方案。
```

你要重点看:

- 用哪个外设
- 用哪些引脚
- 数据从哪里来，到哪里去
- 哪些地方可能不稳定
- 最小验证方法是什么

### Step 3: 让 AI 写最小代码

可以这样问:

```text
请只实现 USART1 接收文本并串口回显，不要加 LCD，不要加 RTOS。
保留现有工程结构，少改无关文件。
```

原则:

- 一次只加一个模块
- 能跑就先跑
- 不追求一开始就漂亮
- 不要同时改太多文件

### Step 4: 自己烧录测试

测试时记录三件事:

```text
我期望看到什么:
实际看到什么:
有没有报错/异常现象:
```

示例:

```text
期望: 串口发送 hello, STM32 回显 hello
实际: 串口没有回显
现象: LED 正常闪烁, 串口助手能打开 COM3
```

### Step 5: 把现象交给 AI 分析

好的提问方式:

```text
现象: LCD 全白，但 LED 会闪，串口能收到 ACK。
硬件: ST7920 12864, PSB 接 GND, PB0 接 RS/CS, PA5 接 E/SCLK, PA7 接 RW/SID。
代码: 这是 lcd_write_byte()。
请按可能性从高到低帮我排查。
```

不要只问:

```text
为什么不行？
```

### Step 6: 跑通后让 AI 解释代码

可以这样问:

```text
请按函数调用顺序解释这段代码。
只讲我必须理解的关键点，不要逐行废话。
```

你要能回答:

- 初始化在哪里
- 启动函数是哪一句
- 中断/回调在哪里
- 数据缓冲区在哪里
- 出错后怎么恢复

### Step 7: 自己做一个小修改

每个模块跑通后，必须做一个小修改。

示例:

- 改 LED 闪烁频率
- 改串口波特率
- 改 LCD 显示内容
- 改 PWM 占空比
- 改按键短按/长按逻辑
- 改 RTOS 任务延时

这一步是为了确认你不是只会烧录，而是真的理解了一点。

## 3. 每个模块的掌握标准

### GPIO

你需要掌握:

- 输入和输出的区别
- 上拉、下拉、浮空是什么意思
- 输出高低电平怎么控制
- 输入按键怎么消抖

最小验证:

```text
按键按下，LED 状态改变。
```

必须能改:

- LED 引脚
- 按键触发电平
- 消抖时间

### UART

你需要掌握:

- TX/RX 要交叉连接
- GND 必须共地
- 波特率两边必须一致
- 阻塞接收、中断接收、DMA 接收的大概区别

常见函数:

```c
HAL_UART_Transmit()
HAL_UART_Receive()
HAL_UART_Receive_IT()
HAL_UART_Receive_DMA()
```

最小验证:

```text
PC 发 hello, STM32 回显 hello。
```

必须能改:

- 波特率
- 接收缓冲区长度
- 回显内容

### SPI / LCD

你需要掌握:

- SCLK、MOSI、MISO、CS 分别是什么
- CS 极性必须确认
- SPI 模式必须确认
- LCD 初始化时序很重要
- 显示异常时先查接线、对比度、复位、CS、延时

最小验证:

```text
LCD 显示 4 行固定文字。
```

必须能改:

- 显示内容
- 刷新周期
- 反色/清屏逻辑

### I2C / 传感器

你需要掌握:

- SDA/SCL 需要上拉
- I2C 地址有 7 位和 8 位写法
- 先读设备 ID 或固定寄存器验证通信
- 传感器读数需要按数据手册换算

最小验证:

```text
串口打印传感器原始数据和换算后的数据。
```

必须能改:

- 设备地址
- 采样周期
- 数据换算公式

### Timer / PWM

你需要掌握:

- 定时器靠频率、预分频、计数周期工作
- PWM 由周期和比较值决定
- 舵机、电机、LED 调光都常用 PWM

常见函数:

```c
HAL_TIM_PWM_Start()
__HAL_TIM_SET_COMPARE()
```

最小验证:

```text
LED 呼吸灯，或者舵机角度变化。
```

必须能改:

- PWM 频率
- 占空比
- 输出通道

### DMA

你需要掌握:

- DMA 是帮 CPU 搬数据
- UART DMA 适合连续收发
- ADC DMA 适合连续采样
- 缓冲区大小和数据边界很重要

最小验证:

```text
UART DMA 接收一帧固定长度数据，收到后回显。
```

必须能改:

- 缓冲区长度
- 帧头
- 超时处理

### FreeRTOS

你需要掌握:

- Task 是多个小 while 循环
- 优先级决定谁更重要
- vTaskDelay 是任务让出 CPU
- Queue 用来传数据
- Semaphore 用来通知事件
- Mutex 用来保护共享资源
- 每个任务都有自己的栈

常见函数:

```c
xTaskCreate()
vTaskStartScheduler()
vTaskDelay()
xQueueSend()
xQueueReceive()
xSemaphoreGive()
xSemaphoreTake()
```

最小验证:

```text
LED 任务闪烁，串口任务收命令，显示任务刷新屏幕，三个任务同时工作。
```

必须能改:

- 任务延时
- 任务优先级
- 队列长度
- 任务栈大小

## 4. 推荐主项目

建议项目:

> STM32 桌面信息屏

这个项目适合你现在的学习路线，因为它能把串口、LCD、按键、RTOS、协议、PC 脚本都串起来。

### 基础功能

- PC 通过串口发送文本
- STM32 接收数据
- 12864 LCD 显示文本或位图
- LED 显示通信状态
- 简单 ACK 防止串口连续发太快

### 进阶功能

- 按键切换页面
- FreeRTOS 分任务
- PC 发送系统监控数据
- 加温湿度传感器
- 加错误状态页面
- 加串口命令配置刷新率/显示模式

### 任务拆分示例

```text
UartRxTask
  接收 PC 数据，解析帧，投递到队列

DisplayTask
  从队列拿显示数据，刷新 LCD

KeyTask
  扫描按键，切换页面或模式

StatusTask
  控制 LED，记录通信状态和错误状态
```

### 数据流

```text
PC
-> UART
-> UartRxTask
-> Queue
-> DisplayTask
-> LCD
```

## 5. 六周执行计划

### 第 1 周: 串口 + LCD 最小系统

目标:

- LCD 显示固定文字
- PC 串口发送 4 行文本
- STM32 显示到 LCD

验收:

- 连续发送 100 次文本不死机
- LCD 不花屏
- 错误接线能通过现象判断

你要理解:

- USART1 的 TX/RX
- LCD 的 CS/SCLK/SID/RST
- 文本帧格式

### 第 2 周: 协议稳定化

目标:

- 加帧头
- 加模式字节
- 加 ACK
- 加超时和错误恢复

验收:

- PC 连续发数据不乱帧
- STM32 忙时不会丢到完全失控
- 串口断开后重新连接能恢复

你要理解:

- 什么是帧同步
- 为什么要 ACK
- 缓冲区为什么不能乱写

### 第 3 周: 按键和页面

目标:

- 加 1 到 3 个按键
- 支持页面切换
- 支持清屏/暂停刷新

验收:

- 按键无明显抖动
- 长按/短按行为清楚
- LCD 页面切换不乱

你要理解:

- GPIO 输入
- 上拉/下拉
- 软件消抖

### 第 4 周: 加入 FreeRTOS

目标:

- 通信、显示、按键拆成任务
- 用队列传显示数据
- 避免多个任务同时操作 LCD

验收:

- 三个任务同时工作
- 串口刷新时按键仍有响应
- LCD 不被多个任务抢着写

你要理解:

- 任务优先级
- 队列
- 互斥锁或单显示任务模型
- 任务栈大小

### 第 5 周: 加传感器或 PC 系统监控

二选一:

方案 A:

- 加温湿度传感器
- LCD 显示本地环境数据

方案 B:

- PC 发送 CPU/内存/时间
- LCD 显示桌面状态

验收:

- 数据周期刷新
- 断开传感器或 PC 不死机
- 错误状态有显示

你要理解:

- 数据采样周期
- 错误处理
- 显示数据和原始数据分离

### 第 6 周: 整理成作品

目标:

- 整理代码结构
- 写 README
- 写接线表
- 写故障排查
- 保留测试脚本

验收:

- 断电重启能正常运行
- 按 README 能重新搭起来
- 常见故障有排查方法

你要理解:

- 项目不是跑一次就结束
- 可维护性也是能力的一部分

## 6. 每周固定问 AI 的问题

每周结束，把当前代码和现象给 AI，然后问:

```text
1. 这个模块我必须理解的 5 个点是什么？
2. 这段代码最容易出 bug 的地方在哪里？
3. 给我 3 个小改动练习，用来验证我真的懂了。
4. 帮我 review 代码，重点找稳定性、边界条件和缺少测试的地方。
```

如果 AI 给你一大堆答案，只抓最重要的 3 到 5 条执行。

## 7. 三份必须保留的笔记

### 模块笔记

示例:

```text
UART:
作用: PC 和 STM32 通信
引脚: PA9 TX, PA10 RX
关键函数: HAL_UART_Transmit, HAL_UART_Receive_DMA
常见坑: TX/RX 接反、波特率不一致、GND 没共地、DMA 缓冲区长度不对
```

### 错误笔记

示例:

```text
现象: LCD 全白
原因: ST7920 串行模式 CS 极性写反
解决: RS/CS 高电平选中，低电平空闲
结论: 以后调 SPI/LCD 必须确认 CS 极性、SPI 模式、初始化延时
```

### 小抄笔记

示例:

```c
HAL_GPIO_WritePin(GPIOC, GPIO_PIN_13, GPIO_PIN_RESET);
HAL_UART_Transmit(&huart1, buf, len, 100);
HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);
__HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_1, duty);
vTaskDelay(pdMS_TO_TICKS(100));
```

## 8. 怎么判断自己真的进步了

你不需要靠背函数判断。

用这些标准判断:

- 能把需求拆成外设和模块
- 能看懂 AI 写出来的关键代码
- 能指出代码里最容易错的地方
- 能自己改一个参数或小逻辑
- 能根据现象说出 3 个可能原因
- 能写出接线表和测试步骤
- 能让项目连续运行一段时间不崩

做到这些，你就不是只会跟教程，而是在真正开发。

## 9. 和 AI 协作的好提示词

### 拆方案

```text
我想做一个 STM32 项目: [描述需求]。
请先不要写代码，帮我拆外设、数据流、模块、风险点和最小验证步骤。
```

### 写最小模块

```text
请只实现 [某一个功能]。
要求:
- 保留现有工程结构
- 不改无关文件
- 给出测试步骤
- 说明关键函数
```

### 代码审查

```text
请以 code review 的方式看这段代码。
优先找 bug、竞态、边界条件、硬件时序、缺少错误处理的问题。
不要直接重写，先列问题。
```

### 调试

```text
现象:
硬件连接:
我已经验证:
相关代码:

请按可能性从高到低列排查步骤。
```

### 学习复盘

```text
这个模块已经跑通。
请告诉我必须理解的 5 个点、常见坑、以及 3 个小改动练习。
```

## 10. 最后提醒

不要追求一开始就完美。

STM32 项目最靠谱的路线是:

```text
先能跑
再稳定
再整理
再扩展
```

你可以少写大段代码，但不能少做验证。

你可以不背 HAL 函数，但不能不知道数据怎么流动。

你可以让 AI 帮你写驱动，但你必须知道接线、时序、初始化、测试方法。

这就是路线二真正的价值: 项目能做出来，人也会变强。
