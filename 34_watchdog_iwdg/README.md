# 第 34 课：IWDG 独立看门狗

## 1. 本课到底在学什么

本课表面现象是：程序启动 IWDG 独立看门狗。按住 PA0 时，程序周期性喂狗，系统正常运行；松开 PA0 后停止喂狗，约 2 秒后 MCU 被 IWDG 强制复位。复位后如果检测到上一次是 IWDG 复位，PC13 LED 会快闪 3 次。

真正要学的是：看门狗不是普通延时器，而是硬件级“程序还活着吗”的监督机制。

本课完整链路是：

```text
IWDG 使用独立 LSI 时钟倒计时
  -> 软件配置 PR/RLR 得到约 2 秒窗口
  -> 软件必须在超时前写 KR=0xAAAA 喂狗
  -> 如果程序卡死或故意不喂狗
  -> IWDG 计数到 0
  -> 硬件强制复位 MCU
  -> RCC->CSR 记录 IWDG 复位标志
  -> 下次启动读标志并用 LED 提示
```

这节课重点不是“让板子重启”，而是理解 IWDG 为什么独立、为什么启动后不能停止、为什么需要明确喂狗策略。

## 2. 本课学习目标

学完本课，你应该能回答：

1. IWDG 是什么，和普通定时器有什么不同？
2. “独立”到底独立在哪里？
3. LSI 为什么让 IWDG 更可靠，但超时时间不够精确？
4. `KR=0x5555`、`0xAAAA`、`0xCCCC` 三个值分别做什么？
5. `PR` 和 `RLR` 如何决定超时时间？
6. 为什么写完 `PR/RLR` 要等待 `SR.PVU/RVU` 清零？
7. 为什么 IWDG 一旦启动软件无法关闭？
8. HAL 版 `HAL_IWDG_Init()` 和 `HAL_IWDG_Refresh()` 分别封装了哪些寄存器动作？

## 4. 实验硬件

- 开发板：STM32F103C8T6 BluePill
- 下载器：ST-Link
- LED：PC13，常见 BluePill 为低电平点亮
- 按键：PA0 接按键，按下接 GND，内部上拉

按键逻辑：

- PA0 按下：读到低电平，程序喂狗
- PA0 松开：读到高电平，程序停止喂狗，等待 IWDG 复位

## 5. 先建立一个最基本的脑图

```text
启动后先读 RCC->CSR.IWDGRSTF
  -> 判断上一次是否 IWDG 复位
  -> 清除复位标志 RMVF
  -> 如果是 IWDG 复位，LED 快闪 3 次

初始化 GPIO
  -> PC13 输出
  -> PA0 上拉输入

初始化 IWDG
  -> KR=0x5555 解锁 PR/RLR
  -> PR=32 分频
  -> RLR=2500
  -> 等 SR.PVU/RVU 清零
  -> KR=0xAAAA 先喂一次
  -> KR=0xCCCC 启动

主循环
  -> 按住 PA0：KR=0xAAAA 喂狗，LED 闪一下
  -> 松开 PA0：不喂狗，LED 常亮
  -> 约 2 秒后硬件复位
```

这里最关键的是：IWDG 启动后，主循环不再拥有“要不要复位”的最终决定权。只要超时没喂狗，硬件会直接复位 MCU。

## 6. 先认识本课里出现的核心名词

### 6.1 `IWDG` 是什么

`IWDG` 是 Independent Watchdog，独立看门狗。

它属于芯片内部安全监督外设。它本质是一个硬件递减计数器，计数到 0 后复位 MCU。

它控制的是系统是否被强制复位。程序必须周期性喂狗，证明主流程仍能运行到安全位置。

### 6.2 `独立` 是什么

IWDG 的“独立”主要指它使用独立的 LSI 时钟，而不是依赖 APB 总线时钟或主系统时钟。

这属于硬件可靠性层。即使 HSE、PLL 或主程序时钟配置异常，IWDG 仍可能继续运行并复位系统。

如果用普通 TIM 模拟看门狗，主时钟坏了，定时器也可能一起失效。

### 6.3 `LSI` 是什么

`LSI` 是 Low Speed Internal，内部低速振荡器，典型频率约 40kHz。

它是 IWDG 的时钟来源。优点是独立，缺点是精度不高，误差可能较大。

所以本课约 2 秒超时只是估算，不是精确 2.000 秒。

### 6.4 `喂狗` 是什么

喂狗就是在 IWDG 超时前重装计数器，让倒计时从 `RLR` 重新开始。

寄存器版写：

```c
IWDG->KR = 0xAAAAU;
```

HAL 版调用：

```c
HAL_IWDG_Refresh(&hiwdg);
```

如果喂狗太晚，IWDG 会复位。喂狗太随意，又可能掩盖程序局部卡死问题。

### 6.5 `IWDG->KR` 是什么

`KR` 是 Key Register，键寄存器。

它不是普通数值寄存器，而是通过写入特定“钥匙值”触发动作：

- `0x5555`：解锁 `PR/RLR` 写保护
- `0xAAAA`：重装计数器，也就是喂狗
- `0xCCCC`：启动 IWDG

写错钥匙值不会达到预期动作。

### 6.6 `IWDG->PR` 是什么

`PR` 是 Prescaler Register，预分频寄存器。

它决定 LSI 先被除以多少再送给 IWDG 计数器。本课设置 PR 对应 32 分频。

分频越大，计数器走得越慢，最大超时时间越长。

### 6.7 `IWDG->RLR` 是什么

`RLR` 是 Reload Register，重装寄存器。

喂狗时，计数器会重新装载 `RLR` 的值。本课 `RLR=2500`。

超时时间近似为：

```text
timeout = RLR / (LSI / prescaler)
        = 2500 / (40000 / 32)
        ≈ 2s
```

### 6.8 `IWDG->SR` 是什么

`SR` 是 Status Register，状态寄存器。

本课关注：

- `PVU`：预分频值正在更新
- `RVU`：重装值正在更新

写完 `PR/RLR` 后要等待这两个位清零，确认配置同步完成。

### 6.9 `IWDG 启动后不能停止` 是什么

IWDG 一旦通过 `KR=0xCCCC` 启动，软件不能关闭它，只能持续喂狗或等待复位。

这属于安全设计：程序跑飞后不能靠错误代码把看门狗关掉。

所以真实项目中，启动 IWDG 前必须确认喂狗策略和超时时间合理。

### 6.10 `RCC->CSR.IWDGRSTF` 是什么

`IWDGRSTF` 是 RCC 控制状态寄存器里的复位来源标志。

如果上一次复位由 IWDG 引起，这个位会置 1。本课启动时读取它，若为 1 就让 LED 快闪 3 次。

### 6.11 `RCC->CSR.RMVF` 是什么

`RMVF` 是 Reset flag clear，复位标志清除位。

复位标志不会自动清除。代码读取完原因后写 `RMVF` 清除，避免下次启动误判。

### 6.12 `IWDG_HandleTypeDef` 是什么

HAL 版使用 `IWDG_HandleTypeDef hiwdg` 描述 IWDG 外设和初始化参数。

`hiwdg.Instance = IWDG` 绑定外设，`hiwdg.Init.Prescaler/Reload` 对应底层 `PR/RLR`。

### 6.13 `HAL_IWDG_Init()` 是什么

这是 HAL 初始化并启动 IWDG 的 API。

它封装 `KR=0x5555` 解锁、写 `PR/RLR`、等待 `SR`、先喂狗、`KR=0xCCCC` 启动。

调用成功后，IWDG 已经开始倒计时。

### 6.14 `HAL_IWDG_Refresh()` 是什么

这是 HAL 喂狗 API。

底层就是向 `IWDG->KR` 写 `0xAAAA`，重装计数器。它不改变 PR/RLR，也不停止 IWDG。

它属于 HAL 工程层，对应寄存器版主循环里的 `IWDG->KR = 0xAAAAU`。调用它的前提是 IWDG 已经初始化并启动，且调用周期必须小于实际超时时间。

这个 API 不能“补救”已经超时的系统。IWDG 计数器到 0 后硬件会复位 MCU，不会等软件再调用刷新函数。因此喂狗点必须放在关键任务确认正常之后，而不是随手放在 while(1) 顶部。

### 6.15 `喂狗策略` 是什么

喂狗策略是工程上决定“什么时候允许刷新看门狗”的规则。好策略应该证明系统关键任务还活着，例如通信任务、采样任务、控制任务都更新了健康标志后，才统一喂狗。

本课用 PA0 人为控制是否喂狗，是为了演示现象。真实产品里如果无条件在主循环喂狗，即使某个关键模块已经卡死，只要主循环还转，看门狗就不会复位，失去了监督意义。

## 7. 寄存器版代码逐步讲解

### 7.1 读取复位来源

```c
uint8_t was_iwdg_reset = (RCC->CSR & RCC_CSR_IWDGRSTF) != 0U;
RCC->CSR |= RCC_CSR_RMVF;
```

先判断上次是否 IWDG 复位，再清除复位标志。顺序不能反，否则读不到原因。

### 7.2 初始化 LED 和按键

PC13 配成推挽输出。PA0 配成上拉输入，按下接 GND，`IDR0=0` 表示按下。

### 7.3 IWDG 复位提示

如果 `was_iwdg_reset` 为真，`led_blink(3)` 快闪三次。这个现象证明上一次重启不是普通上电，而是 IWDG 超时导致。

### 7.4 解锁 PR/RLR

```c
IWDG->KR = 0x5555U;
```

PR 和 RLR 默认写保护，必须先写这个钥匙值解锁。它不是启动，也不是喂狗。

### 7.5 配置 PR

```c
IWDG->PR = IWDG_PR_PR_0 | IWDG_PR_PR_1;
```

该组合对应预分频 32。LSI 约 40kHz，经 32 分频后约 1250Hz。

### 7.6 配置 RLR

```c
IWDG->RLR = 2500U;
```

计数器从约 2500 往下数。按典型 LSI 估算，超时时间约 2 秒。

### 7.7 等待 SR 同步完成

```c
while ((IWDG->SR & (IWDG_SR_PVU | IWDG_SR_RVU)) != 0U) {}
```

IWDG 时钟域和 CPU 时钟域不同，写 PR/RLR 后需要同步。等待 PVU/RVU 清零表示更新完成。

### 7.8 启动前先喂一次

```c
IWDG->KR = 0xAAAAU;
```

这让计数器从完整 RLR 值开始倒计时，避免启动后立即从不确定计数值开始。

### 7.9 启动 IWDG

```c
IWDG->KR = 0xCCCCU;
```

启动后无法软件关闭。之后必须在超时前写 `0xAAAA`。

### 7.10 主循环按键喂狗

按下 PA0 时，代码写 `KR=0xAAAA` 并闪 LED 一次。只要每次循环都能在约 2 秒内执行到这句，系统不会复位。

### 7.11 松开按键停止喂狗

松开 PA0 后代码不写 `0xAAAA`，LED 常亮。IWDG 继续倒计时，到 0 后硬件复位 MCU。

### 7.12 复位后的再次启动

复位后程序从头运行，再次读取 `RCC->CSR.IWDGRSTF`，看到 IWDG 复位标志后快闪 3 次。

## 8. HAL 版代码逐步讲解

### 8.1 `HAL_Init()`

HAL 版先初始化 HAL 基础环境和 SysTick。IWDG 本身不依赖 SysTick，但 LED 闪烁和 `HAL_Delay()` 依赖它。

### 8.2 `__HAL_RCC_GET_FLAG()`

```c
__HAL_RCC_GET_FLAG(RCC_FLAG_IWDGRST)
```

它读取 RCC 复位标志，对应寄存器版读 `RCC->CSR.IWDGRSTF`。

### 8.3 `__HAL_RCC_CLEAR_RESET_FLAGS()`

该宏清除复位标志，对应写 `RCC->CSR.RMVF`。

### 8.4 HAL GPIO 初始化

PC13 配成输出，PA0 配成 `GPIO_PULLUP` 输入。`HAL_GPIO_ReadPin()` 读到 `GPIO_PIN_RESET` 表示按键按下。

### 8.5 `IWDG_HandleTypeDef hiwdg`

`hiwdg.Instance = IWDG` 绑定外设。`Prescaler=IWDG_PRESCALER_32` 对应 PR 分频 32，`Reload=2500` 对应 RLR。

### 8.6 `HAL_IWDG_Init()`

该函数执行解锁、配置 PR/RLR、等待 SR、喂狗、启动 IWDG 的完整流程。调用后 IWDG 已经运行。

这个 API 有返回值，工程中应检查是否为 `HAL_OK`。虽然 IWDG 初始化通常很少失败，但如果参数超出范围或底层状态异常，忽略返回值会让你误以为看门狗已经按预期启动。

更重要的是调用时机：`HAL_IWDG_Init()` 不是“只配置不启动”。一旦调用成功，后面的代码就必须按策略周期性 `HAL_IWDG_Refresh()`，否则系统会被硬件复位。

### 8.7 `HAL_IWDG_Refresh()`

按键按下时调用它喂狗。底层写 `KR=0xAAAA`，把计数器重装到 RLR。

### 8.8 HAL 版主循环

按住 PA0 时每 300ms 左右喂狗并闪 LED；松开时只点亮 LED，不喂狗。超时后 MCU 复位。

## 9. 两个版本真正应该怎么学

寄存器版重点看三个钥匙值：

```text
0x5555 解锁
0xAAAA 喂狗
0xCCCC 启动
```

HAL 版重点看字段映射：

```text
Prescaler -> PR
Reload -> RLR
HAL_IWDG_Init -> 解锁/配置/启动
HAL_IWDG_Refresh -> KR=0xAAAA
```

IWDG 的工程重点是“什么时候喂狗”。API 很简单，但喂狗策略不简单。

## 10. 检验问题清单

### 10.1 IWDG 为什么比普通定时器更适合做看门狗？

**答**：IWDG 使用独立 LSI 时钟，不依赖主系统时钟。主时钟或程序异常时，它仍可能继续工作并复位系统。

### 10.2 `KR=0x5555` 是喂狗吗？

**答**：不是。它是解锁 PR/RLR 写保护，让软件可以配置预分频和重装值。

### 10.3 真正喂狗写哪个值？

**答**：写 `KR=0xAAAA`，HAL 版对应 `HAL_IWDG_Refresh()`。

### 10.4 `KR=0xCCCC` 做什么？

**答**：启动 IWDG。启动后软件无法关闭，只能喂狗或等待复位。

### 10.5 本课超时时间为什么只是约 2 秒？

**答**：因为 LSI 典型约 40kHz，但精度不高。PR=32、RLR=2500 按典型值估算约 2 秒，实际会有偏差。

### 10.6 为什么要清 `RCC->CSR.RMVF`？

**答**：复位标志不会自动清除。如果不清，后续启动可能继续读到旧的 IWDG 复位标志。

### 10.7 松开 PA0 后为什么会复位？

**答**：松开后程序不再喂狗，IWDG 计数器持续递减到 0，硬件触发 MCU 复位。

### 10.8 HAL 初始化 IWDG 后还能停止吗？

**答**：不能。HAL 初始化内部会启动 IWDG，一旦启动软件无法停止。

## 11. 工程实现步骤

### 11.1 需求分析

本课要演示“正常喂狗不复位”和“停止喂狗会复位”。用 PA0 控制是否喂狗，用 PC13 显示状态和复位原因。

### 11.2 硬件核查

确认 PC13 LED 正常，PA0 按键按下接 GND，未按下通过内部上拉为高电平。

### 11.3 寄存器路线

先读复位来源并清标志，再初始化 GPIO，配置 IWDG 的 PR/RLR，等待同步，喂一次狗，启动 IWDG，在主循环按条件刷新。

### 11.4 HAL 路线

用 RCC 宏读清复位标志，用 GPIO HAL 配 LED/按键，用 `IWDG_HandleTypeDef` 填 Prescaler/Reload，调用 `HAL_IWDG_Init()` 和 `HAL_IWDG_Refresh()`。

### 11.5 工程思维

看门狗应该在系统确认关键任务正常后喂，而不是在任意循环无条件喂。否则主任务卡死但喂狗代码还活着，会掩盖故障。

### 11.6 常见工程陷阱

启动 IWDG 后想关闭、超时时间按精确 LSI 计算、忘记等待 SR 同步、忘记清复位标志、喂狗点放得太随意，都会让工程行为不可靠。

还有一个常见陷阱是调试断点。IWDG 使用独立 LSI，CPU 停在断点时看门狗仍可能继续计数，时间一到就复位。调试看门狗代码时，要么缩短断点停留，要么先临时不启动 IWDG。

另一个陷阱是把 2 秒当成精确定时。LSI 误差很大，实际超时时间可能明显偏离估算值。工程设计应给喂狗周期留足裕量，例如预计 2 秒超时，就不要把喂狗周期设计成 1.9 秒。

## 12. 运行现象

按住 PA0 上电或复位后，程序会周期性喂狗，LED 闪烁，系统不复位。

松开 PA0 后 LED 常亮，约 2 秒后 MCU 复位。复位后如果检测到 IWDG 复位标志，LED 会快闪 3 次。

## 13. 常见问题排查

### 13.1 松开按键后不复位

检查 IWDG 是否真正启动，寄存器版是否写了 `KR=0xCCCC`，HAL 版是否调用 `HAL_IWDG_Init()`。

### 13.2 按住按键仍然复位

检查 PA0 输入逻辑是否正确，按下是否读到低电平，喂狗周期是否小于 IWDG 超时时间。

### 13.3 复位后没有快闪 3 次

检查是否在清标志前读取 `IWDGRSTF`。如果先清 `RMVF`，就读不到复位原因。

### 13.4 超时时间和 2 秒差很多

LSI 本来精度不高，这是 IWDG 特性。需要更准确的监督时间时，要留足裕量或校准 LSI。

计算值只用于估算：`40kHz / 32` 得到约 1250Hz，`2500 / 1250` 约 2 秒。但如果 LSI 实际是 30kHz 或 50kHz，超时时间就会明显变化。设计喂狗周期时必须按最坏情况留余量。

### 13.5 调试时程序总是重启

IWDG 启动后调试断点也可能导致停止喂狗而复位。调试看门狗程序时要缩短断点停留时间，或先暂时不启动 IWDG。

## 14. 本课最核心的结论

1. IWDG 是独立看门狗，倒计时到 0 会硬件复位 MCU。
2. IWDG 使用 LSI，独立可靠但超时时间不精确。
3. `KR` 的三个钥匙值分别用于解锁、喂狗和启动。
4. `PR/RLR` 决定超时时间，写完后要等待 `SR` 同步完成。
5. IWDG 一旦启动，软件无法关闭。
6. `RCC->CSR` 可以判断上一次是否 IWDG 复位。
7. 工程上喂狗策略比 API 调用本身更重要。

## 15. 建议你现在怎么读这节课

先把 `KR` 三个值背后的动作分清楚，再按第 5 章画出复位前后流程。最后对照 HAL 版，确认 `HAL_IWDG_Init()` 内部并不是魔法，而是封装了同一串寄存器写入。

1. 把 `RLR` 改成 1250，观察超时时间大约减半。
2. 把喂狗周期改到接近超时时间，观察 LSI 误差带来的不稳定。
3. 增加串口或调试变量记录复位次数。
4. 设计一个“主任务正常才喂狗”的标志位机制。
