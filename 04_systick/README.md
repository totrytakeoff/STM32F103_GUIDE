# 第 4 课：SysTick 毫秒节拍

## 1. 本课到底在学什么

前面几课你已经做过两种延时方式：

- 最原始的空转延时循环
- HAL 版里直接调用 `HAL_Delay()`

但如果你之前没有专门学过 `SysTick`，那么 `HAL_Delay()` 对你来说其实还是个黑盒。

这节课就是要把这个黑盒拆开。

本课真正学习的是：

- `SysTick` 是什么
- 它为什么能每隔固定时间触发一次
- 什么叫“1ms 节拍”
- 为什么 `HAL_Delay()` 能工作
- 自己写一个 `delay_ms()` 的最小链路是什么

也就是说，这节课不是在学“又一种闪灯”，而是在学：

- 时间基准是怎么建立出来的

这个知识后面会直接影响：

- 定时器
- 中断
- RTOS 的 Tick
- 非阻塞任务调度
- 超时判断

## 5. 先建立一个最基本的脑图

这节课的核心链路如下：

1. 先把系统时钟配置到一个已知频率，比如 72MHz
2. 再配置 `SysTick`，让它每 1ms 产生一次事件
3. `SysTick` 产生中断后，进入 `SysTick_Handler()`
4. 在中断里把一个全局毫秒计数变量加 1
5. `delay_ms()` 通过比较这个计数变量，判断时间是否到了
6. 时间到了以后，主循环继续执行

最关键的理解点是：

- `SysTick` 本身不是“延时函数”
- 它只是一个能周期性打点的硬件计数器

真正的延时是你基于这个“周期性打点”自己建立出来的。

## 6. 先认识本课里出现的核心名词

### 6.1 SysTick 是什么

`SysTick` 是 Cortex-M 内核自带的系统定时器。

注意这句话很重要：

- 它是“内核级”的定时器
- 不是某个普通片上外设定时器

所以它和我们后面会学到的 `TIM2`、`TIM3` 这种通用定时器不完全是一类东西。

你可以先把它理解成：

- CPU 内部自带的一个专门用来做系统节拍的简单定时器

### 6.2 为什么 SysTick 适合做 1ms 节拍

因为它的设计初衷之一就是：

- 提供统一的系统时间基准

它非常适合拿来做：

- 毫秒 Tick
- 简单延时
- 操作系统节拍

### 6.3 `SysTick->LOAD` 是什么

这是 SysTick 的重装值寄存器。

你可以理解成：

- 每次数到 0 之前，总共要数多少拍

例如：

- 如果系统时钟是 72MHz
- 你希望 1ms 触发一次

那么 1ms 内的时钟数是：

- `72,000,000 / 1000 = 72,000`

因为计数器通常从设定值一直减到 0，所以常写成：

- `72000 - 1`

### 6.4 `SysTick->VAL` 是什么

这是 SysTick 当前计数值寄存器。

你可以把它理解成：

- 这个定时器当前还剩多少没数完

### 6.5 `SysTick->CTRL` 是什么

这是 SysTick 控制寄存器。

它主要负责：

- 选择时钟源
- 开启/关闭计数器
- 开启/关闭 SysTick 中断
- 反映计数是否到过 0

所以你会在寄存器版看到它被用来打开 SysTick。

### 6.6 `SysTick_Handler()` 是什么

这是 SysTick 中断服务函数。

当 SysTick 计数到 0，且中断开启后，CPU 就会跳到这个函数里执行。

所以你可以把它理解成：

- “每次 SysTick 到点后自动执行的函数”

### 6.7 `g_ms_ticks` 是什么

这是我们自己定义的一个软件变量。

它不是硬件寄存器，也不是 HAL 自带的东西。

它的作用是：

- 记录“已经过去了多少个毫秒”

如果 `SysTick_Handler()` 每 1ms 进入一次，那么我们每次在里面做：

- `g_ms_ticks++`

就等于建立了一个“软件毫秒时钟”。

### 6.8 `HAL_SYSTICK_Config()` 是什么

这是 HAL 提供的一个 SysTick 配置函数。

它的作用可以先理解成：

- 帮你设置 SysTick 的重装值
- 帮你启动 SysTick

本质上，它仍然是在配置 SysTick 相关寄存器。

### 6.9 `HAL_IncTick()` 是什么

这是 HAL 内部用于维护时间基准的函数。

你可以先理解成：

- 每次 SysTick 中断到了，HAL 自己也要把内部的毫秒计数加 1

这就是为什么 `HAL_Delay()` 能用的核心前提之一。

### 6.10 `HAL_Delay()` 为什么能工作

它能工作，不是因为它自己会“制造时间”，而是因为：

1. SysTick 已经被配置成固定周期中断
2. 每次中断时，HAL 的内部 Tick 会增加
3. `HAL_Delay()` 通过比较这个内部 Tick，判断时间有没有到

所以：

- 没有 SysTick 时间基准，`HAL_Delay()` 就失去基础

## 8. 寄存器版代码逐步讲解

对应代码文件：

- [reg/src/main.c](/home/myself/workspace/mcu/stm32/STM32F103C8T6/04_systick/reg/src/main.c)

### 8.1 寄存器版主线

寄存器版会按这个顺序做：

1. 配系统时钟到 72MHz
2. 初始化 LED 输出
3. 配置 SysTick 为 1ms 中断一次
4. 在中断函数里维护毫秒计数
5. 用 `delay_ms()` 控制 LED 周期闪烁

### 8.2 为什么 `LOAD = 72000 - 1`

因为在 72MHz 下：

- 每秒有 72,000,000 个时钟周期
- 1ms 是 1/1000 秒

所以 1ms 对应的时钟数是：

- `72,000,000 / 1000 = 72,000`

而 SysTick 是从设定值减到 0，所以通常写：

- `72,000 - 1`

### 8.3 `delay_ms()` 为什么能成立

因为 `g_ms_ticks` 每 1ms 增加一次。

所以：

- 记录一个开始时间
- 一直等到“当前时间 - 开始时间 >= 目标毫秒数”

就相当于实现了一个基于系统节拍的阻塞延时。

## 9. HAL版代码逐步讲解

对应代码文件：

- [hal/src/main.c](/home/myself/workspace/mcu/stm32/STM32F103C8T6/04_systick/hal/src/main.c)

### 9.1 HAL 版这节课的重点不是“会用 HAL_Delay”

真正重点是：

- 知道 `HAL_Delay()` 为什么能用

### 9.2 `HAL_SYSTICK_Config()` 做了什么

它本质上做的是：

- 设置重装值
- 打开 SysTick
- 建立固定周期触发

### 9.3 为什么 `SysTick_Handler()` 里还要调用 `HAL_IncTick()`

因为 HAL 自己的延时和时间基准依赖它的内部 Tick 计数。

如果你把 SysTick 中断接管了，但忘了调用：

- `HAL_IncTick()`

那么可能会出现：

- `HAL_Delay()` 不正常

### 9.4 为什么 HAL 版里还要保留自定义计数器

因为这节课的目标不是单纯依赖 HAL，而是要让你同时看见：

- HAL 内部的 Tick
- 你自己建立的 Tick

这样你才会真正理解：

- HAL 并不是“凭空延时”

## 12. 常见问题排查

### 12.1 LED 完全不闪

优先检查：

1. 系统时钟是否真的配置成功
2. SysTick 是否真的开启
3. 中断是否真的在进入 `SysTick_Handler()`
4. LED 引脚是否已正确初始化

### 12.2 `delay_ms()` 一直卡住

优先检查：

1. `g_ms_ticks` 是否在中断中递增
2. `SysTick_Handler()` 是否函数名写对
3. SysTick 中断是否开启

### 12.3 HAL 版中 `HAL_Delay()` 不工作

优先检查：

1. 是否调用了 `HAL_Init()`
2. 是否正确配置了 SysTick
3. `SysTick_Handler()` 里是否调用了 `HAL_IncTick()`

## 13. 本课要点总结

本课最核心的结论如下：

1. `SysTick` 是 Cortex-M 内核自带的系统定时器
2. 它非常适合建立统一的毫秒节拍
3. `LOAD` 决定周期长度，`CTRL` 负责启动和中断控制
4. 在 72MHz 下，1ms 周期常用重装值是 `72000 - 1`
5. `SysTick_Handler()` 周期执行后，就能建立软件时间基准
6. `HAL_Delay()` 的基础是 SysTick 和 HAL 内部 Tick 维护
