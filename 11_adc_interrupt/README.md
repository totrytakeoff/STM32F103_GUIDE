# 第 11 课：ADC 中断采样

## 1. 本课到底在学什么

上一课你已经学会了：

- 单通道 ADC 轮询采样

也就是：

1. 启动一次 ADC 转换
2. 然后 CPU 一直等
3. 等到 `EOC` 置位
4. 再读取结果

这种方法简单直接，但也有一个明显问题：

- **CPU 在等待转换期间什么都不做**

如果转换需要 21us，这 21us 内 CPU 不能做其他任务——在 72MHz 下，21us ≈ 1512 条指令被浪费了。

这节课我们要切换到另一种方式：

- **ADC 转换完成后主动发中断**

也就是说，本课真正要学的是：

- ADC 也能像定时器、EXTI 一样用中断
- `EOC` 不仅能被轮询，也能作为中断触发条件
- 为什么中断方式能减少"傻等"
- ADC 的中断处理链路是什么
- HAL 的 ADC 中断回调机制是怎么组织的
- 为什么中断中要再次启动下一次转换（形成持续采样链）

### 轮询 vs 中断 对比

| 特性 | 轮询 | 中断 |
|------|------|------|
| CPU 等待方式 | 循环检查 EOC，被占用 | 转换期间 CPU 可以干其他事 |
| 响应方式 | 主动去问"转完了吗？" | 转完了通知 CPU |
| 代码位置 | 主循环中直接调用 | 写在中断函数/回调中 |
| 实时性 | 取决于主循环调度 | 中断触发后立即响应 |
| 适用场景 | 简单、低频、单任务 | 需要"后台采样"或"实时响应" |

## 5. 先建立一个最基本的脑图

本课完整链路如下：

```
main 中第一次启动转换
       │
       ▼
ADC 开始工作
       │
       ▼
ADC 完成转换 → EOC 置位 → EOCIE 已使能
                               │
                               ▼
                        产生中断请求
                               │
                               ▼
                    NVIC 总闸是否打开？
                        是        否
                        │        （进不了中断）
                        ▼
          CPU 跳转到 ADC1_2_IRQHandler()
                        │
                        ▼
                    读取 DR
                    控制 LED
                    再次启动转换 ← 形成循环
```

1. 把 `PA1` 配成模拟输入
2. 配置 ADC1：通道、采样时间、校准
3. **打开 ADC 的"转换完成中断"（EOCIE）**
4. **在 `NVIC` 中使能 ADC 中断**
5. 软件触发一次转换
6. ADC 内部完成转换后，把 `EOC` 置位
7. 因为中断已开启，所以 CPU 进入 `ADC1_2_IRQHandler()`
8. 在中断处理里读取结果
9. 用结果更新 LED
10. **再次启动下一次转换**

这节课最重要的理解点是：

- **轮询模式**：CPU 主动盯着 ADC 看
- **中断模式**：ADC 完成后主动通知 CPU

## 6. 先认识本课里出现的核心名词

### 6.1 什么是 ADC 转换完成中断

ADC 每完成一次转换，都会得到一个结果。

如果只用轮询，那么程序会反复去问：

- 转完了吗？

而中断模式则是：

- 转完后 ADC 自己发通知

这个通知的核心依据就是：

- **EOC 标志 + EOCIE 使能位**

### 6.2 `EOC` 在中断模式下扮演什么角色

你上一课已经知道：

- `EOC = End Of Conversion`

它表示：

- 一次转换已经完成

在轮询模式下，你是自己去检查它。

在中断模式下，它还会进一步作为：

- **触发 ADC 中断请求的依据**

当 `EOC` 置位且 `EOCIE = 1` 时，硬件自动发出中断请求。

### 6.3 `EOCIE` 是什么

这是 ADC 的转换完成中断使能位，位于 `CR1` 寄存器的 bit 5。

把它打开后，就表示：

- 当 `EOC` 置位时
- ADC 可以发出中断请求

如果不打开它，就算转换完成了，也不会自动进中断。

### 6.4 为什么 ADC 中断还要配 NVIC

这和定时器中断、EXTI 中断是同样的道理。

ADC 自己能产生中断请求，只是第一层。

要让 CPU 真正响应，还必须：

- 在 `NVIC` 中使能 ADC 对应的 IRQ

所以 ADC 中断同样有两层：

1. **ADC 外设侧**（EOCIE）：允许 ADC 发出中断请求
2. **NVIC 侧**（NVIC_EnableIRQ）：允许该中断请求到达 CPU 内核

### 6.5 `ADC1_2_IRQHandler()` 是什么

在 STM32F103 上：

- `ADC1` 和 `ADC2`

共用一个中断入口函数：

- `ADC1_2_IRQHandler()`

所以当 ADC1 转换完成中断发生时，CPU 会进这里。

### 6.6 为什么中断里通常要读 `DR`

因为：

- 转换结果就在 `DR`

如果不读，数据就一直留在 DR 寄存器里，下次转换会覆盖它。

而且读取 `DR` 后，`EOC` 标志会自动清除（F103 的特性）。

### 6.7 为什么中断里还会再次启动下一次转换

因为本课采用的是：

- 单次转换（ContinuousConvMode = DISABLE）
- 软件触发

这意味着：

- 一次转换结束后，不会自动无限继续

所以如果你希望程序持续采样，就可以在中断处理后：

- **再启动下一次转换**

这是一种很典型的"中断驱动连续单次采样"写法。

### 6.8 中断与主循环如何配合

```
时间轴：
main            ADC 硬件         中断
 │                │                │
 ├─ Start_IT ───→ │                │
 │  （继续执行）    │                │
 │  主循环做其他事  │  正在转换       │
 │                ├── 转换完成 ───→ │
 │                │                ├─ 读 DR
 │                │                ├─ 控制 LED
 │                │                ├─ Start_IT
 │                │←────────────── │
 │                │  正在转换       │
 │  主循环继续     ├── 转换完成 ───→ │
 │                │                └─ ...
```

### 6.9 HAL 的三段 ADC 中断链路是什么

HAL 版本课最核心的链路是：

| 步骤 | HAL 函数 | 作用 |
|------|---------|------|
| 1 | `HAL_ADC_Start_IT()` | 启动 ADC + 使能 EOC 中断 |
| 2 | `ADC1_2_IRQHandler()` → `HAL_ADC_IRQHandler()` | 中断入口 → HAL 分发器 |
| 3 | `HAL_ADC_ConvCpltCallback()` | **你写业务逻辑的地方** |

你可以这样理解：

- `Start_IT`：启动并打开 ADC 中断流程
- `HAL_ADC_IRQHandler`：HAL 的通用中断分发器（检测标志、清除标志）
- `Callback`：你真正写采样完成业务逻辑的地方（读值、处理、再启动）

### 8.1 寄存器版主线

1. 配系统时钟
2. 配 `PC13`
3. 配 `PA1` 模拟输入
4. 配 ADC1：通道、采样时间、校准

6. **使能 `ADC1_2_IRQn`（新！）**
7. 软件启动第一次转换
8. 在 `ADC1_2_IRQHandler()` 里读取结果
9. 用结果更新 LED
10. **再启动下一次转换（新！）**

### 8.2 与上一课（轮询版）的区别总结

| 方面 | 轮询版（10_adc_polling） | 中断版（本课） |
|------|------------------------|---------------|
| 启动转换 | 每次循环都启动 | 第一次在 main，后续在中断 |
| 等待完成 | `while(!(SR & EOC))` | 不等待，由中断通知 |
| 中断使能 | 未使能（EOCIE = 0） | **使能（EOCIE = 1）** |
| NVIC | 未配置 | **配置了** |
| 读取结果 | 在 `adc1_read_channel1()` 中 | 在 `ADC1_2_IRQHandler()` 中 |
| CPU 占用 | 等待期间被占用 | 转换期间 CPU 可处理其他任务 |

### 8.3 关键寄存器速查

| 寄存器 | 位/字段 | 本课值 | 含义 |
|--------|---------|--------|------|
| `ADC1->CR1` | EOCIE | 1 | **使能转换完成中断（新）** |
| `ADC1->CR2` | EXTTRIG + SWSTART | 1 | 软件触发转换 |
| `ADC1->SR` | EOC | 1→读DR自动清 | 转换完成标志 |
| `NVIC->ISER` | ADC1_2_IRQn | 1 | **使能 NVIC 中断通道（新）** |

## 9. HAL 版代码逐步讲解

对应代码文件：

- [hal/src/main.c](/home/myself/workspace/mcu/stm32/STM32F103C8T6/11_adc_interrupt/hal/src/main.c)

### 9.1 HAL 版为什么不再调用 `HAL_ADC_PollForConversion()`

因为这节课要强调的是：

- **中断完成通知**

如果还继续轮询，就失去了本课主线。

### 9.2 `HAL_ADC_Start_IT()` 和 `HAL_ADC_Start()` 的区别是什么

| 函数 | 做了什么 |
|------|---------|
| `HAL_ADC_Start()` | 只启动转换，不使能中断 |
| `HAL_ADC_Start_IT()` | 启动转换 + 自动清除标志 + **使能 EOC 中断** |

`HAL_ADC_Start_IT()` 内部对应寄存器操作：

1. 清除 EOC 等标志位
2. `CR1 |= EOCIE`（使能中断）
3. 启动转换（SWSTART）

### 9.3 为什么业务逻辑写在 `HAL_ADC_ConvCpltCallback()` 里

因为 HAL 试图把：

- 通用中断处理流程

和：

- 用户采样完成后的业务处理

分开。

所以最自然的业务落点就是：

- 转换完成回调

这就是"框架 + 回调"的典型设计模式。

### 9.4 为什么回调里还要再次调用 `HAL_ADC_Start_IT()`

原因和寄存器版一样——本课配置的是"单次转换"模式。

如果不再次启动，ADC 只转一次就停了。

在回调末尾调用 `HAL_ADC_Start_IT()` 就形成了：

```
开始 → 转换完成 → 中断 → 回调 → 再开始 → 转换完成 → 中断 → ...
```

## 10. 寄存器版和 HAL 版对比

### 10.1 寄存器版重点看什么

- `EOCIE` 位在 `CR1` 中
- `ADC1_2_IRQHandler()` 中断函数的写法
- 中断里读取 `DR`
- 中断里再次启动下一次转换

### 10.2 HAL 版重点看什么

- `HAL_ADC_Start_IT()` 启动中断采样
- `ADC1_2_IRQHandler()` → `HAL_ADC_IRQHandler()` → `ConvCpltCallback` 的三层链路
- 回调函数中写业务逻辑

### 10.3 两者如何一一对应

| 功能 | 寄存器版 | HAL 版 |
|------|---------|--------|
| 开 EOC 中断 | `CR1 |= EOCIE` | `Start_IT()` 内部完成 |
| 中断入口 | 直接写 `ADC1_2_IRQHandler` | 在其中调用 `HAL_ADC_IRQHandler` |
| 中断处理 | 直接在中断函数中写 | `HAL_ADC_IRQHandler` 分发到回调 |
| 读结果 | `DR` | `HAL_ADC_GetValue()` |
| 再次启动 | `CR2 |= EXTTRIG \| SWSTART` | `HAL_ADC_Start_IT()` |
| NVIC 配置 | `NVIC_EnableIRQ()` | `HAL_NVIC_EnableIRQ()` |

## 11. 运行现象

程序正常时，旋转电位器会看到：

- 当采样值高于阈值，LED 亮
- 当采样值低于阈值，LED 灭

但这一次背后的采样机制已经不是轮询，而是：

- ADC 转换完成后主动中断通知

肉眼看到的现象可能和上一课一模一样，但背后的 CPU 行为完全不同——转换期间 CPU 不再傻等，可以去做其他任务。

## 12. 常见问题排查

### 12.1 ADC 完全不进中断

优先检查：

1. **是否打开了 `EOCIE`**——没有它 ADC 不会产生中断请求
2. **是否使能了 `ADC1_2_IRQn`**——NVIC 总闸没开也不行
3. **是否真的启动了第一次转换**——初始化完成后是否调了 Start 或 SWSTART
4. **ADC 校准是否已完成**——CAL 位是否回到 0
5. **是否在 NVIC 中设置了正确的优先级**——如果优先级被更高优先级中断屏蔽也可能不进

### 12.2 中断函数进了，但值不对

优先检查：

1. 通道是否仍然是通道 1
2. `PA1` 是否仍然配置成模拟模式
3. 电位器接线是否正确

### 12.3 程序只响应一次

优先检查：

1. **是否在中断处理完后重新启动了下一次转换**
2. 如果是寄存器版：`adc1_start_conversion()` 是否被调用
3. 如果是 HAL 版：回调末尾是否有 `HAL_ADC_Start_IT()`

## 13. 本课要点总结

本课最核心的结论如下：

1. **`EOC` 不仅能被轮询，也能触发中断**——只需使能 EOCIE 位
2. **ADC 中断同样需要"外设侧 + NVIC 侧"双层配置**——缺一不可
3. **中断方式能避免主循环一直傻等**——CPU 在转换期间可处理其他任务
4. **单次软件触发模式下，若想持续采样，必须要在中断处理中再次启动转换**
5. **HAL 的 ADC 中断流程本质上仍然在围绕 `EOC` 和结果寄存器运作**
6. **HAL 用"回调"把底层中断处理和用户业务逻辑分离开**
7. **"启动 → 完成 → 中断 → 再启动"形成一个持续采样链**

### 13.1 中断使能的两层配置

```
ADC 外设侧（外设内部允许）       NVIC 侧（CPU 内核允许）
    CR1.EOCIE = 1          +      NVIC_EnableIRQ()
          │                           │
          ▼                           ▼
    ADC 可以产生中断请求          CPU 可以接收中断请求
          │                           │
          └───────────┬───────────────┘
                      ▼
              中断到达 CPU 内核
              进入 ADC1_2_IRQHandler()
```
