# STM32 教学约束 Prompt

## 一、用途说明

这份文档用于约束后续整个 STM32 教学过程中的讲解风格、结构、粒度和输出要求。

它的目标很明确：

- 以后所有课程都必须像 `01_led` 这一课一样讲
- 不能只给“能跑的代码”，必须把“为什么这样写”讲清楚
- 不能让寄存器、宏、结构体、API 突然出现又不解释
- 不能只讲理论，必须落到可以直接上板运行的 Demo

这份文档既是教学风格总结，也是后续课程编写时的统一 Prompt 规范。

## 二、LED Demo 已验证出的教学要点总结

`01_led` 这一课已经验证了当前最适合你的讲解方式。它的核心教学要点如下。

### 1. 先讲“这节课真正学什么”

不能一上来就丢代码。

必须先明确：

- 这节课表面现象是什么
- 真正要学的底层链路是什么
- 这节课在整个 STM32 学习体系里处于什么位置

以 `LED` 为例，表面上是在点灯，但真正学习的是：

- GPIO 控制链路
- 外设时钟使能
- 引脚模式配置
- 输出寄存器控制
- HAL 和寄存器版的对应关系

### 2. 先建立“完整脑图”，再进入代码

用户不是看不懂代码本身，而是看不懂代码为什么会突然出现这些名字。

所以在正式讲代码前，必须先给出：

- 这个功能的执行链路
- 每一步依赖的硬件模块
- 数据或控制信号如何流动

例如 `LED` 课中，必须先建立这条链路：

1. 先开 GPIO 外设时钟
2. 再配置引脚模式
3. 再输出电平
4. 最后 LED 才会亮灭

只有先有了这条链路，后面的寄存器和 API 才不会像“空穴来风”一样突然冒出来。

### 3. 每个新出现的名词都必须解释“它是谁、干什么、为什么现在出现”

这是当前最重要的教学规则。

以后凡是第一次出现下面这类东西，都必须单独解释：

- 外设模块名
- 寄存器名
- 位定义宏
- 结构体
- HAL API
- 回调函数
- 中断号
- 标志位

而且解释时必须包含下面四个点：

1. 它是什么
2. 它是干什么的
3. 为什么此时必须用它
4. 它在整个执行链路里处于哪一步

例如不能只写：

```c
RCC->APB2ENR |= RCC_APB2ENR_IOPCEN;
```

而不解释：

- `RCC` 是什么
- `APB2ENR` 是什么
- `IOPCEN` 是什么
- 为什么 GPIOC 一定要先开时钟

### 4. 寄存器版必须逐步拆解，不允许“黑箱式位运算”

用户当前最大的障碍，不是不会写代码，而是看不懂寄存器操作为什么这么写。

所以以后寄存器版的讲解必须做到：

- 解释寄存器名字的含义
- 解释寄存器为什么属于这个外设
- 解释为什么这个引脚落在这个寄存器里
- 解释为什么要先清位再设位
- 解释位掩码宏对应的是哪几个 bit
- 解释最终配置值在硬件上代表什么意思

例如：

- 为什么 `PC13` 配在 `CRH`，不是 `CRL`
- 为什么 `MODE13 = 10` 表示输出模式 2MHz
- 为什么 `CNF13 = 00` 表示通用推挽输出
- 为什么 `BSRR` 用来置高，`BRR` 用来置低

### 5. HAL 版不能只讲“会用”，必须讲它和底层的映射关系

以后 HAL 版不能写成“调用几个 API 就结束”。

每个 HAL API 都必须讲清楚：

- 它的输入参数是什么
- 这些参数分别代表什么硬件配置
- 它本质上帮你封装了哪一步底层操作
- 它和寄存器版哪一段逻辑是一一对应的

例如在 `LED` 课中，必须明确：

- `__HAL_RCC_GPIOC_CLK_ENABLE()` 本质是开 GPIOC 时钟
- `HAL_GPIO_Init()` 本质是配置 GPIO 模式寄存器
- `HAL_GPIO_WritePin()` 本质是控制 GPIO 输出电平

### 6. 代码注释必须以“教学注释”为目标，而不是简单说明语法

后续代码中的注释，不能只写这种低价值内容：

- 给变量赋值
- 调用某某函数

而应该写：

- 为什么需要这一步
- 这一步依赖前面的哪个条件
- 这个参数选择背后的硬件原因是什么
- 如果少了这一步会出现什么问题

也就是说，注释的重点必须是：

- 原因
- 目的
- 位置
- 后果

而不是只重复代码表面意思。

### 7. 课程文档必须先解释，再讲代码，再做对比，再讲排错

不能把课程文档写成单纯的代码附录。

一节课的文档必须有完整教学闭环：

1. 本课到底学什么
2. 学习目标
3. 硬件环境
4. 先建立脑图
5. 核心名词解释
6. 寄存器版逐步讲解
7. HAL版逐步讲解
8. 两种方式对比
9. 运行现象
10. 常见问题排查
11. 本课结论总结
12. 扩展练习

### 8. 第一课只引入当前必须的知识，不允许超前塞太多内容

这个点也很重要。

例如 `LED` 课里没有把：

- 时钟树
- SysTick
- 定时器中断
- 更复杂的启动流程

一起塞进来，这是正确的。

以后每一课都必须遵守：

- 只讲完成当前 Demo 所必须的知识
- 但对新出现的知识必须讲透

也就是：

- 控制“广度”
- 保证“深度”

### 9. 运行现象和故障排查必须是文档固定部分

用户不是只需要代码，还需要知道：

- 正常现象应该是什么
- 如果结果不对，先查什么
- 哪一步最容易错

所以每一课都必须有：

- 运行现象说明
- 常见错误
- 排查顺序

### 10. 寄存器版和 HAL 版必须同时存在，并互相映射

后续每一课都必须保留两套方案：

- `reg/`
- `hal/`

而且讲解中必须持续强调：

- 它们做的是同一件事
- 只是抽象层次不同
- HAL 不是脱离硬件的魔法

## 三、后续课程统一教学 Prompt

下面这部分是后续所有 STM32 课程编写时必须遵守的 Prompt 约束。

可把它视为统一教学模板。

### 0. 绝对基准：必须按 `01_led`、`02_gpio_key`、`03_clock_tree`、`04_core_architecture_memory_map` 的颗粒度写

`01_led/README.md`、`02_gpio_key/README.md`、`03_clock_tree/README.md`、`04_core_architecture_memory_map/README.md` 不是“风格参考”而已，而是后续课程 README 的最低详细程度基准。

后续每一课都必须像前 4 课一样做到：

- 第 6 章不能只列项目符号，必须用 `6.1`、`6.2`、`6.3` 这种小节逐个讲名词
- 每个名词不能只写一句定义，必须讲“它是谁、名字怎么拆、属于哪个硬件/软件层、负责什么、为什么本课会出现、写错会有什么现象”
- 第 7 章不能只概括“寄存器版按顺序读”，必须像 `01_led` 一样按函数、按语句、按寄存器位逐步拆
- 第 8 章不能只说“HAL 封装了寄存器”，必须逐个 API、逐个结构体字段讲它到底对应哪一步寄存器配置
- 第 10 章问题必须带答案，不能只列问题
- 第 11 章必须有需求分析、硬件核查、寄存器路线、HAL 路线、工程思维、常见工程陷阱，不能只写编译下载步骤
- 第 13 章排错必须按具体现象拆小节，不能只写几条泛泛的排查建议

如果某一课写出来比 `01`、`02`、`03`、`04` 明显短、明显粗、明显像模板，就视为不合格，必须返工。

前 4 课已经证明，合格的教学文档应该具备下面这种层层推进的密度：

- `01_led` 的层次：从“LED 亮灭”拆到 GPIOC 时钟、PC13 在 CRH、MODE/CNF、BSRR/BRR、HAL GPIO 结构体字段。
- `02_gpio_key` 的层次：从“按键按下”拆到 PA0 接 GND、内部上拉、CRL、MODE0/CNF0、ODR0 选择上拉、IDR 读取、HAL `GPIO_PULLUP` 对应底层两步。
- `03_clock_tree` 的层次：从“系统跑 72MHz”拆到 HSE、PLL、SYSCLK、FLASH 等待周期、RCC->CR、RCC->CFGR、HSERDY/PLLRDY/SWS、APB1/APB2 分频、HAL 两个 RCC 结构体。
- `04_core_architecture_memory_map` 的层次：从“Watch 里看到地址”拆到 Cortex-M3 地址访问、Flash/SRAM/Peripheral 基地址、GPIOC_BASE、SCB->CPUID、volatile、CMSIS 宏和结构体指针。

也就是说，每一课都必须把“肉眼现象”一路拆到“硬件层、寄存器层、bit 层、C 代码层、HAL 封装层、排错层”。少一层都不算讲透。

### 0.1 每一课必须按六层拆解

写任何课程 README 时，都必须显式覆盖下面六层。

第一层：现象层。

- 学生能看到什么现象，例如 LED 亮灭、串口输出、PWM 波形、ADC 数值变化、任务切换。
- 这个现象由哪个引脚、变量、外设状态或调试窗口体现。
- 现象正常时应该是什么样，异常时最常见会变成什么样。

第二层：物理/硬件层。

- 外部器件怎么接，例如按键接 GND、LED 低电平点亮、PA0 接 PA6、串口 TX/RX 交叉。
- 引脚为什么选这个，例如 PA0 是 TIM2_CH1，PA6 是 TIM3_CH1，PA8 是 TIM1_CH1。
- 电平默认状态是什么，例如上拉后松开为 1，按下接地为 0。
- 如果接线反了、没共地、没接外设，会出现什么具体现象。

第三层：芯片模块层。

- 当前课用到哪些模块：RCC、GPIO、AFIO、EXTI、TIM、ADC、USART、DMA、NVIC、SysTick、SCB、FreeRTOS 内核对象等。
- 每个模块在链路中负责哪一步。
- 模块之间如何传递控制或数据，例如 GPIO 电平进入 EXTI，TIM 捕获把 CNT 锁存到 CCR，DMA 从外设搬到内存。

第四层：寄存器/bit 层。

- 每个关键寄存器为什么出现，属于哪个模块，控制哪类行为。
- 每个关键 bit 或字段的值为什么这样选。
- 必须写清楚配置值来源，例如 `72000 - 1`、`1000 - 1`、`0xFFFF`、`PLL x9`、`APB1 /2`。
- 必须解释清位、置位、等待标志、清标志、启动外设的顺序。

第五层：C 代码/CMSIS 层。

- `GPIOC->CRH`、`TIM2->PSC`、`SCB->CPUID` 这类写法本质是结构体指针访问固定地址。
- 宏名要拆开讲，例如 `RCC_APB2ENR_IOPCEN` 可以读出“RCC 的 APB2ENR 里 GPIOC 使能位”。
- `volatile`、全局变量、中断共享变量、回调函数参数都要说明为什么这样写。

第六层：HAL/工程层。

- HAL 结构体字段如何表达寄存器配置意图。
- HAL API 最终封装了哪几步底层动作。
- 工程中为什么常用 HAL，学习时为什么先看寄存器。
- 后续维护、排错时应该观察哪些寄存器、变量、返回值和波形。

每一课的第 1、5、6、7、8、10、11、13、14 章都必须体现这六层，不能只在某一章随便提一下。

### 0.2 每章最低细致程度

为了避免“标题齐了但内容粗”，后续 README 还必须满足下面的最低细致程度。

- 第 1 章至少 3 段：表面现象、真实底层链路、这课与前后课程的关系。
- 第 2 章至少 7 个具体学习目标，必须是学生能自测的问题，不写空话。
- 第 5 章必须有一段箭头链路或编号链路，且每一步都能对应到后面代码。
- 第 6 章通常不少于 8 个小节；简单课可以少一点，但只要出现新寄存器/API/结构体/标志位，就必须单独讲。
- 第 7 章通常不少于 8 个小节；必须覆盖 `main()`、每个初始化函数、每个关键寄存器组、主循环或中断处理。
- 第 8 章通常不少于 8 个小节；必须覆盖 HAL 初始化顺序、每个结构体、每个关键字段、每个启动 API、每个回调。
- 第 10 章必须 6~10 个问题且带答案，答案必须能直接帮助排错或解释代码。
- 第 11 章必须有 11.1 到 11.6，不能合并。
- 第 13 章必须 3~5 个具体现象型小节，例如“LED 完全不亮”“周期不对”“中断只进一次”“HAL_Delay 卡住”。
- 第 14 章必须是 4~8 条完整结论，不能只写一句“核心链路是 XXX”。

这不是鼓励灌水，而是要求把每一层真正讲出来。

### 0.3 章节之间必须互相对应

README 不能每章各说各话，必须形成闭环：

- 第 1 章提出的真实链路，必须在第 5 章展开。
- 第 5 章链路里的每个模块，必须在第 6 章有名词解释。
- 第 6 章解释过的寄存器/API，必须在第 7/8 章回到代码里逐步讲。
- 第 7/8 章讲过的关键步骤，必须在第 10 章变成检验问题。
- 第 10 章问到的易错点，必须在第 13 章给排查路径。
- 第 14 章结论必须回扣第 1 章的真实链路。

如果某个寄存器/API 只在代码里出现，文档没有解释；或者第 6 章解释了一个名词，第 7/8 章没有落回代码，都视为不合格。

### 0.4 写 README 前必须先做代码盘点

写任何一课 README 前，必须先完整阅读并盘点：

- `reg/src/main.c`
- `hal/src/main.c`
- `platformio.ini`
- 本课涉及的头文件、宏定义或 FreeRTOS 配置文件

然后把本课代码中的内容分成下面几类，确保文档全部覆盖：

1. 硬件资源：
   - 用了哪些引脚
   - 每个引脚接了什么
   - 每个引脚是输入、输出、复用、模拟还是中断
   - 需要观察的是 LED、电平、波形、串口输出、调试变量还是任务状态
2. 外设模块：
   - 用了哪些 STM32 外设
   - 每个外设挂在哪条总线
   - 为什么要先开对应 RCC 时钟
   - 外设之间有没有连接关系，例如 AFIO 映射、TIM 通道、DMA 请求、EXTI 线
3. 寄存器和 bit：
   - 代码里出现的每个寄存器
   - 代码里出现的每个位掩码宏
   - 每个配置值的来源
   - 每个等待标志、清除标志和启动位
4. HAL 对象：
   - 每个 `xxx_HandleTypeDef`
   - 每个 `xxx_InitTypeDef`
   - 每个 HAL API
   - 每个 HAL 回调函数
   - 每个宏，例如 `__HAL_RCC_xxx_CLK_ENABLE()`、`__HAL_TIM_SET_COMPARE()`
5. 软件状态：
   - 全局变量
   - `volatile` 变量
   - 中断和主循环共享变量
   - FreeRTOS 任务句柄、队列句柄、信号量句柄、事件组、软件定时器等

盘点出来的内容必须在 README 第 5、6、7、8、10、13 章中全部落地。不能代码里出现了 `TIM_DIER_CC1IE`，文档却只泛泛说“打开中断”。

### 0.5 第 6 章名词解释必须回答的固定问题

第 6 章每个 `6.X` 小节都必须回答下面这些问题。可以用自然段写，不一定机械列标题，但内容必须齐。

1. 它的名字是什么，英文全称或可拆解含义是什么？
2. 中文通常叫什么？
3. 它属于哪一层？
   - Cortex-M 内核
   - STM32 片上外设
   - RCC/总线/时钟树
   - GPIO 引脚电气层
   - HAL 软件封装
   - FreeRTOS 内核对象
   - C 语言/CMSIS 抽象
4. 它控制什么？
   - 时钟是否打开
   - 引脚模式
   - 数据输入/输出
   - 计数周期
   - 比较值/占空比
   - 中断使能
   - 标志位
   - DMA 搬运
   - 任务阻塞/唤醒
5. 本课为什么需要它？
6. 它在第 5 章链路中处于哪一步？
7. 代码里哪里使用它？要尽量点出函数名或语句。
8. 如果它配错、漏掉、顺序错，会出现什么具体现象？
9. 它和前面课程学过的哪个概念相似或不同？

例如讲 `GPIO_PULLUP`，不能只写“内部上拉”。必须写清楚：

- 它是 HAL 层字段值
- 它表达的是 GPIO 输入默认电平策略
- 在 F103 底层对应 `CNF = 10` 加 `ODR = 1`
- 本课按键接 GND，所以松开必须默认读 1
- 漏掉后 PA0 可能悬空，LED 乱闪或按键状态不稳定

例如讲 `FLASH->ACR`，不能只写“Flash 访问控制寄存器”。必须写清楚：

- 它属于 Flash 接口，不是 RCC
- 它控制 CPU 高频读 Flash 时的等待周期和预取
- 72MHz 前必须先配 `LATENCY_2`
- 漏掉后可能切到高速后运行不稳定或跑飞

### 0.6 第 7 章寄存器版必须按“语句后果”写

第 7 章不是把代码复制一遍，也不是概括函数功能，而是要讲每条关键语句执行后的硬件后果。

遇到下面类型的语句，必须这样讲：

1. 开时钟：
   - 写的是哪个 RCC 使能寄存器
   - 这个外设挂在哪条总线
   - 为什么不开时钟后续配置不可靠或无效
2. 清配置位：
   - 清的是哪个字段
   - 这个字段占几 bit
   - 为什么要保留其他引脚/其他字段
   - 如果不清，旧配置会怎样影响新配置
3. 设配置位：
   - 设置后的二进制字段值是什么
   - 这个值在参考手册里代表什么模式
   - 为什么本课需要这个模式
4. 写数字配置：
   - 数字从哪里来
   - 是否有 `+1/-1` 规则
   - 和时钟频率、周期、占空比、波特率、采样时间有什么关系
5. 等待标志：
   - 等的是哪个 ready/busy/complete 标志
   - 硬件什么时候置位或清零
   - 如果不等会发生什么
6. 清标志：
   - 标志由谁置位
   - 是写 0 清、写 1 清，还是读写序列清
   - 不清会导致重复中断、无法继续发送、读到旧状态还是其他现象
7. 启动外设：
   - 哪一位是真正启动位
   - 启动前必须完成哪些配置
   - 启动后硬件开始自动做什么
8. 中断函数：
   - 中断源从哪里来
   - 外设内部使能位是什么
   - NVIC 中断号是什么
   - ISR 里为什么先判断标志
   - ISR 里为什么要尽量短
9. 主循环：
   - 主循环读哪个变量、寄存器或标志
   - 这个值由谁更新
   - 为什么这部分不放在中断里

第 7 章必须让学生读完后能做到：遮住代码，只看 README，也能说出寄存器版初始化顺序和每一步的硬件后果。

### 0.7 第 8 章 HAL 版必须按“字段到寄存器”映射写

第 8 章必须把 HAL 层拆开，不能只写“HAL 简化了配置”。

每个 HAL 结构体必须逐字段解释：

- 字段名是什么
- 字段值是什么
- 字段表达的硬件意图是什么
- 底层大致对应哪个寄存器或字段
- 如果字段写错，现象是什么

例如定时器 PWM 课程里：

- `htim.Init.Prescaler` 必须对应 `PSC`
- `htim.Init.Period` 必须对应 `ARR`
- `sConfigOC.Pulse` 必须对应 `CCR`
- `sConfigOC.OCMode` 必须对应 `OCxM`
- `HAL_TIM_PWM_Start()` 必须说明打开通道输出和启动计数器

每个 HAL API 必须说明：

- 调用前必须已经完成什么
- 这个 API 会写哪些类型的寄存器
- 它是否只初始化、不启动
- 它是否会打开中断
- 它是否会清标志
- 它失败时如何处理返回值

每个 HAL 回调必须说明：

- 回调不是用户主动调用的普通函数
- 它由哪个 IRQHandler 里的哪个 HAL 分发函数调用
- 触发它的硬件事件是什么
- 参数代表哪个外设或通道
- 为什么业务代码写在这里

第 8 章必须让学生读完后能把 HAL 代码翻译成寄存器路线，而不是只会照抄 API。

### 0.8 FreeRTOS 课程必须额外拆 RTOS 层，但不能污染普通外设课程

普通外设课程不要乱提 FreeRTOS。只有 FreeRTOS 课程才需要额外讲：

- 任务是什么
- 调度器是什么
- tick 是什么
- 阻塞等待是什么
- 队列、信号量、互斥量、事件组、任务通知分别是什么
- heap 和 stack 分别负责什么
- ISR API 和普通 API 为什么不同
- 中断优先级为什么会影响 RTOS API

FreeRTOS 课程也必须按前面六层拆解，但需要多加一层 RTOS 层：

```text
硬件事件
  -> 中断或轮询
  -> RTOS 对象
  -> 任务阻塞/唤醒
  -> 任务函数处理
  -> 现象输出
```

讲 `Queue` 这类概念时，必须像前 4 课讲 `RCC`、`GPIOA->CRL` 那样拆：

- 它属于 FreeRTOS 内核对象，不属于 STM32 硬件外设
- 它控制的是任务之间的数据传递和阻塞唤醒关系
- `xQueueCreate()` 创建对象
- `xQueueSend()` 往对象里放数据，可能唤醒接收任务
- `xQueueReceive()` 从对象取数据，队列空时任务可能阻塞
- `portMAX_DELAY` 表示在允许无限阻塞的条件下一直等
- 队列满会导致发送失败或发送任务阻塞
- 任务解耦是让生产者和消费者不直接互相调用

不能把 RTOS 对象写成“硬件寄存器”，也不能把硬件外设写成“任务对象”。

### 0.12 禁止把多个关键动作压在一行

教学代码中，禁止把多个关键寄存器动作、HAL 配置动作、RTOS API 动作压在同一行。

下面这种写法不合格：

```c
GPIOC->CRH &= ~(GPIO_CRH_MODE13 | GPIO_CRH_CNF13); GPIOC->CRH |= GPIO_CRH_MODE13_1; GPIOC->BSRR = GPIO_BSRR_BS13;
```

必须拆成多行，并配教学注释：

```c
/* PC13 在 CRH 中配置。先清 MODE13/CNF13，避免旧模式残留影响新配置。 */
GPIOC->CRH &= ~(GPIO_CRH_MODE13 | GPIO_CRH_CNF13);

/* MODE13 = 10，表示输出速度 2MHz；CNF13 = 00，表示通用推挽输出。 */
GPIOC->CRH |= GPIO_CRH_MODE13_1;

/* BluePill 板载 LED 通常低电平点亮，所以先输出高电平让 LED 熄灭。 */
GPIOC->BSRR = GPIO_BSRR_BS13;
```

这类注释必须解释：

- 为什么先清位再置位
- 配置字段代表什么硬件模式
- 这一步改变了哪个寄存器或外设状态
- 少了或写错会出现什么现象

### 0.13 禁止把函数压缩成一行

教学代码中，禁止把普通函数、任务函数、Hook 函数、ISR、`main()` 压缩成一行。

下面这种写法不合格：

```c
static void led_toggle_pc13(void){ if((GPIOC->ODR & GPIO_ODR_ODR13)!=0U) GPIOC->BRR=GPIO_BRR_BR13; else GPIOC->BSRR=GPIO_BSRR_BS13; }
```

必须拆开写，让学生能看清楚判断和硬件后果：

```c
static void led_toggle_pc13(void)
{
    /*
     * 先读 ODR 判断 PC13 当前输出状态。
     * 注意：BluePill 板载 LED 常见为低电平点亮，所以电平和亮灭可能相反。
     */
    if ((GPIOC->ODR & GPIO_ODR_ODR13) != 0U) {
        /* 当前输出高电平，写 BRR 把 PC13 拉低。 */
        GPIOC->BRR = GPIO_BRR_BR13;
    } else {
        /* 当前输出低电平，写 BSRR 把 PC13 拉高。 */
        GPIOC->BSRR = GPIO_BSRR_BS13;
    }
}
```

`main()` 必须按步骤展开：

1. 初始化系统时钟。
2. 初始化 GPIO 和外设。
3. 创建 RTOS 对象或初始化业务状态。
4. 创建任务或启动外设。
5. 检查创建/初始化结果。
6. 启动调度器或进入主循环。
7. 说明调度器启动后正常为什么不会再返回。

下面这种写法不合格：

```c
int main(void){ system_clock_72mhz_init(); gpio_init(); g_bin=xSemaphoreCreateBinary(); vTaskStartScheduler(); while(1){} }
```

### 0.14 教学注释必须解释原因、后果、前置条件、错误现象

代码注释不能只重复语法。

不合格注释：

- `初始化 GPIO`
- `创建任务`
- `发送信号量`
- `延时 1 秒`
- `打开时钟`

合格注释必须说明：

- 为什么现在要做这一步
- 这一步依赖前面哪个条件
- 这一步改变了哪个寄存器、外设状态、HAL handle 或 RTOS 对象状态
- 少了这一步会怎样
- 写错后会出现什么具体现象

寄存器版 `reg/src/main.c` 必须能看出底层链路：

- 开 RCC 时钟
- 配 GPIO 模式
- 配外设寄存器
- 清位/置位
- 配置数值来源
- 等待标志
- 清除标志
- 启动外设
- 主循环或中断如何处理结果

HAL 版 `hal/src/main.c` 必须能看出字段到寄存器的映射：

- `HandleTypeDef` 是哪个外设的软件句柄
- `InitTypeDef` 每个字段表达什么硬件意图
- `HAL_xxx_Init()` 大致写了哪类寄存器
- `HAL_xxx_Start()` 是否只是启动，还是也打开中断或 DMA
- Callback 由哪个 IRQHandler 或 HAL 分发函数触发

### 0.15 FreeRTOS 代码必须额外解释 RTOS 层

FreeRTOS 课程中，只要代码出现下面内容，就必须有教学注释或邻近说明：

- `xTaskCreate`
- `vTaskStartScheduler`
- `vTaskDelay`
- `xSemaphoreCreateBinary`
- `xSemaphoreCreateCounting`
- `xSemaphoreGive`
- `xSemaphoreTake`
- `xQueueCreate`
- `xQueueSend`
- `xQueueReceive`
- `xEventGroupCreate`
- `xTaskNotifyGive`
- `ulTaskNotifyTake`
- `portMAX_DELAY`
- `pdTRUE` / `pdFALSE` / `pdPASS`
- `taskDISABLE_INTERRUPTS`
- `vApplicationMallocFailedHook`
- `vApplicationStackOverflowHook`
- FromISR 系列 API
- 队列、信号量、互斥量、事件组、任务通知、软件定时器

RTOS 对象创建不能只写代码，必须解释对象含义。

例如：

```c
g_bin = xSemaphoreCreateBinary();
g_count = xSemaphoreCreateCounting(2, 2);
```

必须说明：

- 二值信号量是什么
- 计数信号量是什么
- 最大计数 `2` 代表什么资源数量
- 初始计数 `2` 代表什么初始状态
- 创建失败通常说明 heap 不够
- 为什么创建后必须检查 NULL

任务函数不能压缩成一行。

下面这种写法不合格：

```c
static void giver(void *arg){ (void)arg; while(1){ xSemaphoreGive(g_bin); vTaskDelay(pdMS_TO_TICKS(1000)); } }
```

必须拆开，并说明：

- `arg` 为什么暂时不用
- `while (1)` 为什么是任务常见结构
- `xSemaphoreGive()` 改变哪个 RTOS 对象状态，可能唤醒谁
- `vTaskDelay()` 为什么让任务进入阻塞态
- `pdMS_TO_TICKS()` 为什么要把毫秒转换为 tick

### 0.16 FreeRTOS Hook 函数不能裸写

FreeRTOS Hook 函数不能只写成关中断死循环。

下面这种写法不合格：

```c
void vApplicationMallocFailedHook(void){ taskDISABLE_INTERRUPTS(); while(1){} }
void vApplicationStackOverflowHook(TaskHandle_t task, char *task_name){ (void)task; (void)task_name; taskDISABLE_INTERRUPTS(); while(1){} }
```

必须拆开，并解释：

- Hook 由 FreeRTOS 内核调用，不是用户主动调用
- `MallocFailedHook` 代表 heap 分配失败
- `StackOverflowHook` 代表任务栈溢出
- 为什么这里要关中断，防止系统继续调度造成更多混乱
- 为什么进入死循环等待调试器停住
- `task` 和 `task_name` 参数为什么可能暂时只做 `(void)` 消除告警

- stack
- FromISR

- 阻塞态/就绪态
- RTOS 对象

非 FreeRTOS 课程必须按硬件外设、寄存器、HAL、主循环、中断、DMA、协议状态机来解释。

只有 FreeRTOS 课程才允许把任务状态、heap、stack、FromISR、调度器作为教学内容。

```text
`PSC` 是预分频寄存器。它和 ARR 一起决定周期。
```

合格写法必须接近下面这种颗粒度：

```text
### 6.X `TIM2->PSC` 是什么

`PSC` 全称可以理解成 Prescaler，中文叫预分频寄存器。

它属于 TIM2 这个定时器外设，不属于 GPIO，也不属于 RCC。

它的作用是：
- 把 TIM2 输入时钟先除以 `PSC + 1`
- 给后面的 CNT 计数器提供更慢、更容易使用的计数节拍

在本课里，TIM2 输入时钟是 72MHz，代码写 `PSC = 7200 - 1`，所以 CNT 实际按 10kHz 递增。

如果这里写错：
- 写太小：计数太快，LED 翻转周期变短
- 写太大：计数太慢，LED 翻转周期变长
- 忘记写：定时器按默认分频工作，后面的 ARR 计算全部失效
```

### 4. 寄存器讲解规则

寄存器版代码必须做到：

- 逐步讲解寄存器含义
- 逐步讲解位操作目的
- 逐步讲解配置值来源
- 逐步讲解为什么先清零再设置
- 逐步讲解为什么改这个寄存器，而不是另一个寄存器

不能只写“把某位设置为 1”，必须进一步说明：

- 这位控制了什么功能
- 为什么要设置成这个值
- 设置错误会导致什么现象

寄存器版第 7 章必须按照 `01_led` 的方式拆成多个小节，最低要求如下：

- 先讲 `main()` 的完整执行顺序
- 再按函数逐个讲，例如 `system_clock_72mhz_init()`、`gpio_init()`、`tim_init()`、`irq_handler()`
- 每个函数内部必须按关键语句讲，不允许只总结功能
- 遇到 `&= ~MASK` 必须解释为什么先清位，清的是哪几 bit，不清会残留什么旧配置
- 遇到 `|= MASK` 必须解释置位后硬件行为怎么变
- 遇到计算值，例如 `72000 - 1`、`1000 - 1`、`0xFFFF`，必须写出计算来源
- 遇到中断，必须讲清“外设标志 -> 外设中断使能 -> NVIC -> IRQHandler -> 清标志”的完整链路
- 遇到输入/输出引脚，必须说明为什么是 `CRL` 还是 `CRH`，为什么是输入/输出/复用/上拉/浮空

如果只是写：

```text
寄存器版先开时钟，再配置 GPIO，再启动外设。
```

这是不合格的，因为它没有教会学习者看懂寄存器代码。

### 5. HAL 讲解规则

HAL 版代码必须做到：

- 解释每个 API 的作用
- 解释每个参数的含义
- 解释结构体字段对应的硬件含义
- 指出它在底层等价于哪一步寄存器配置

禁止把 HAL 当成黑盒调用。

HAL 第 8 章必须像 `01_led` 一样逐项拆：

- `HAL_Init()` 为什么在前面，它依赖什么，会影响什么
- 每个 `__HAL_RCC_xxx_CLK_ENABLE()` 本质对应哪个 RCC 使能位
- 每个 `xxx_InitTypeDef` 结构体是什么，为什么先 `{0}` 清零
- 每个结构体字段都要解释，例如 `Pin`、`Mode`、`Pull`、`Speed`、`Prescaler`、`Period`、`Pulse`、`OCMode`
- 每个 `HAL_xxx_Init()` 到底把结构体字段写进了哪些寄存器或哪类寄存器
- 每个 `HAL_xxx_Start()` 和 `HAL_xxx_Start_IT()` 必须区分：普通启动、通道启动、中断启动分别打开了哪些底层开关
- 每个回调函数必须说明是谁调用它、什么时候调用、参数代表什么、用户代码为什么写在这里

不能只写“这个 API 封装了底层配置”。必须明确封装了哪一步。

### 6. 注释规则

代码必须带详细中文注释。

注释重点必须放在：

- 为什么这么做
- 这一步在硬件上意味着什么
- 这一步依赖什么前置条件
- 如果不做会怎样

不能只用注释重复代码字面含义。

### 7. 文档结构规则

每个知识点目录下都必须有详细文档，且 README 只能按第四部分规定的 **17 章结构**执行。

旧的“至少包含若干项”只能作为历史说明，不能再作为后续课程的执行标准。

也就是说，后续课程不能只写 14 个大概栏目，也不能只把标题补齐后写成短模板。真正合格的 README 必须：

- 使用第 1 章到第 17 章的统一结构
- 达到 `01_led`、`02_gpio_key`、`03_clock_tree`、`04_core_architecture_memory_map` 的最低颗粒度
- 在第 6、7、8、10、11、13 章把 `reg/src/main.c`、`hal/src/main.c`、`platformio.ini` 和相关配置文件中出现的内容全部落地
- 保证 README 讲到的引脚、外设、协议、波特率、DMA 通道、中断来源、运行现象与代码一致

如果 README 仍然只是“本课学什么、运行现象、常见问题、总结、扩展练习”这种压缩结构，就视为不合格。

### 8. Demo 规则

每一课必须有真实可运行 Demo。

要求：

- 一个知识点一个目录
- 每个目录下同时包含 `reg/` 和 `hal/`
- 两个版本都必须能编译通过
- 尽量保证都能直接上板验证

禁止只写伪代码或只给理论不落地。

### 9. 范围控制规则

每节课只讲完成当前 Demo 必需的内容。

但是：

- 当前课中第一次出现的新概念，必须讲透

不要在一节课中无节制引入后续太多内容。

### 10. 对比规则

每节课都必须明确对比：

- 寄存器版做了什么
- HAL版做了什么
- 两者本质上是如何对应的
- 两种方式分别适合什么场景

### 11. 排错规则

每节课必须明确说明：

- 正常现象应该是什么
- 常见故障有哪些
- 推荐按什么顺序排查
- 最值得观察的寄存器或状态位有哪些

### 12. 输出风格规则

输出必须满足：

- 全中文
- 面向初学者但不敷衍
- 细致、直接、能落地
- 不省略关键解释
- 不用“这里你先记住”来替代应有解释，除非当前层次确实还没展开条件
- 不使用万能模板句
- 不跨课乱塞概念，例如普通外设课程不要突然提 FreeRTOS 的 heap、stack、FromISR
- 不把“硬件层、HAL 层、RTOS 层”的概念混在一起讲
- 不用概括性结论替代代码级解释

### 13. 代码风格规则

代码要满足：

- 尽量简洁
- 但教学注释必须充分
- 变量命名尽量直观
- 函数职责清楚
- 适合被拿来逐行阅读

代码不能只追求“能跑”。后续每一课的 `reg/src/main.c` 和 `hal/src/main.c` 都必须同时满足“能跑、能读、能逐行讲清楚为什么、能让初学者把代码和硬件链路对应起来”。

写代码前和写 README 前都必须同时检查：

- `reg/src/main.c`
- `hal/src/main.c`
- `reg/platformio.ini`
- `hal/platformio.ini`
- 如有 `FreeRTOSConfig.h`，也必须检查

禁止把多个关键动作压在一行里。下面这种写法不合格：

```c
GPIOC->CRH &= ~(GPIO_CRH_MODE13 | GPIO_CRH_CNF13); GPIOC->CRH |= GPIO_CRH_MODE13_1; GPIOC->BSRR = GPIO_BSRR_BS13;
```

必须拆成多行，并用教学注释说明：为什么先清 `MODE/CNF`，`MODE13_1` 代表什么输出模式，为什么 PC13 初始化时先写高电平，BluePill 的 PC13 LED 为什么通常低电平亮。

禁止把函数压缩成一行。下面这种写法不合格：

```c
static void led_toggle_pc13(void){ if((GPIOC->ODR & GPIO_ODR_ODR13)!=0U) GPIOC->BRR=GPIO_BRR_BR13; else GPIOC->BSRR=GPIO_BSRR_BS13; }
```

必须拆开写，让学习者能看清：先读 `ODR` 判断当前输出状态，再根据当前状态写 `BRR` 或 `BSRR`，以及 PC13 电平和 LED 亮灭为什么可能相反。

禁止把 `main()` 写成一整行。`main()` 必须按步骤展开：

1. 初始化系统时钟
2. 初始化 GPIO
3. 初始化外设或创建 RTOS 对象
4. 创建任务或启动外设
5. 检查初始化/创建结果
6. 启动主循环或调度器
7. 解释调度器启动后为什么正常不会再返回

FreeRTOS 课程必须额外解释 RTOS 层。只要代码出现下面内容，就必须有教学注释：

- `xTaskCreate`
- `vTaskStartScheduler`
- `vTaskDelay`
- `xSemaphoreCreateBinary`
- `xSemaphoreCreateCounting`
- `xSemaphoreGive`
- `xSemaphoreTake`
- `xQueueCreate`
- `xQueueSend`
- `xQueueReceive`
- `xEventGroupSetBits`
- `xTaskNotifyGive` / `vTaskNotifyGiveFromISR`
- `portMAX_DELAY`
- `pdTRUE` / `pdPASS`
- `taskDISABLE_INTERRUPTS`
- `vApplicationMallocFailedHook`
- `vApplicationStackOverflowHook`
- `FromISR` 系列 API

FreeRTOS Hook 函数不能裸写。必须解释：

- Hook 由 FreeRTOS 内核调用，不是用户主动调用
- `MallocFailedHook` 代表 heap 分配失败
- `StackOverflowHook` 代表任务栈溢出
- 为什么这里要关中断
- 为什么进入死循环等待调试器
- `task` 和 `task_name` 参数为什么可能暂时只做 `(void)` 消除告警

RTOS 对象创建不能只写代码，必须解释对象含义。例如：

```c
g_bin = xSemaphoreCreateBinary();
g_count = xSemaphoreCreateCounting(2, 2);
```

必须解释二值信号量是什么、计数信号量是什么、最大计数和初始计数分别表示什么、创建失败通常说明 heap 不够、为什么创建后必须检查 `NULL`。

任务函数不能压缩成一行。必须说明：

- `arg` 为什么暂时不用
- `while (1)` 为什么是任务常见结构
- `xSemaphoreGive` 或 `xQueueSend` 会影响哪个 RTOS 对象
- `vTaskDelay` 为什么让任务进入阻塞态
- `pdMS_TO_TICKS` 为什么要把毫秒转成 tick

寄存器版代码必须能让学生看出底层链路：开 RCC 时钟、配置 GPIO 模式、配置外设寄存器、清位/置位、配置数值来源、等待标志、清除标志、启动外设、主循环或中断如何处理结果。

HAL 版代码必须能让学生反推到底层寄存器意图：每个 Handle 是什么，每个 `InitTypeDef` 是什么，每个字段表达什么硬件意图，`HAL_xxx_Init()` 写了哪类寄存器，`HAL_xxx_Start()` 是否只是启动还是也打开中断/DMA，Callback 由哪个 IRQHandler 或 HAL 分发函数触发。

普通外设课程不能乱提 FreeRTOS。只有 FreeRTOS 课程才允许讲任务、heap、stack、FromISR、调度器、阻塞态和就绪态。

### 14. 检验问题清单规则

每一节课必须包含"检验问题清单"独立小节，不可省略。

它的作用是：

- 让学习者看完后能自我验证是否真的理解了
- 防止"觉得看懂了但实际讲不清楚"

要求：

- 列出 6~10 个本课核心问题
- 问题必须覆盖：概念理解、寄存器/API 为什么这样配、顺序依赖、现象反了怎么排查
- 每个问题必须附带标准答案
- 答案要简要但不能敷衍，关键知识点不能省略
- 不能只问"这是什么"，还必须问"为什么"和"如果不对怎么办"
- 参考 01_led 第 10 节的格式和深度

典型问题类型示例：

- "为什么这个寄存器要先清零再设置？"
- "为什么这个引脚不是配在 CRL 而是 CRH？"
- "如果现象和预期相反，第一步应该查什么？"
- "寄存器版和 HAL 版本质上有什么区别？"
- "如果忘了开外设时钟会怎样？"

禁止：

- 只列问题不给答案
- 问题太少（少于 6 个）
- 问题流于表面（如只问"这是什么"）

### 15. 工程实现步骤与工程思维规则

每一节课必须包含"工程实现步骤"独立小节，不可省略。

它的作用是：

- 让学习者不只学"代码怎么写"，还能知道"在真实项目里这东西应该怎么用"
- 建立从学习到工程的映射

要求：

1. 先拆解实现路线：
   - 寄存器实现路线：按正确顺序列出每一大步，说明为什么顺序不能乱
   - HAL 实现路线：按正确顺序列出每一大步，说明和寄存器路线的对应关系
2. 再说明工程思维：
   - 学习阶段应该怎么学（为什么先学寄存器版再看 HAL 版）
   - 工程项目中应该怎么用（什么时候用寄存器、什么时候用 HAL）
   - 长期维护时应该关注什么
3. 最后列出常见工程陷阱：
   - 列举 3~5 个本课最容易犯的真实工程错误
   - 说明错误现象和正确做法
   - 要来源于真实踩坑经验，而不是纯理论推导

禁止：

- 只写步骤不写"为什么顺序必须这样"
- 把工程思维写成空话套话
- 不写常见工程陷阱

参考 01_led 第 11 节的格式和深度。

### 16. 最终目标规则

每一节课结束后，学习者不仅要能"把程序跑起来"，还要能自己回答：

- 为什么需要这些寄存器
- 为什么需要这些 API
- 为什么顺序不能乱
- 这节课最小可运行链路是什么
- 如果坏了该先查哪里

只有达到这个程度，这一课才算真正学会。

## 四、课程标准章节模板

基于 `01_led` 已验证的教学效果，后续所有课程都必须严格遵循以下 **17 章结构**。这是统一的课程模板。

### 第 1 章：本课到底在学什么

目的：让学习者明确知道这节课表面现象和深层目标。

必须包含：

- 这节课表面现象是什么
- 真正学习的底层链路是什么
- 这节课在整个 STM32 学习体系中的位置
- 这个知识点为什么重要、后面会用到哪里

风格参考：01_led 第 1 章

### 第 2 章：本课学习目标

目的：给出具体的"学完后你应该能回答的问题"。

必须包含：

- 6~10 个具体问题
- 这些问题应该涵盖：概念、原理、寄存器/API、顺序依赖、排错
- 问题之间要有逻辑顺序，从简到难

风格参考：01_led 第 2 章

### 第 3 章：本课目录结构

目的：让学习者知道代码文件的组织方式。

必须包含：

```
XX_coursename/
├── README.md
├── reg/
│   ├── platformio.ini
│   └── src/main.c
└── hal/
    ├── platformio.ini
    └── src/main.c
```

附带说明：

- `reg/` 的用途
- `hal/` 的用途

风格参考：01_led 第 3 章

### 第 4 章：实验硬件

目的：明确这个 Demo 需要什么硬件。

必须包含：

- 开发板型号
- 下载器
- 外部接线（如有）
- 板载资源说明
- 前置假设（如"系统时钟需要配置成 72MHz"）

风格参考：01_led 第 4 章

### 第 5 章：先建立一个最基本的脑图

目的：在讲代码之前，先让学习者知道整体流程。

必须包含：

- 用序列号或箭头展示实现链路
- 每一步说明依赖什么、做什么、导致什么
- 标出每一步对应的硬件模块或软件动作
- 指出整条链路中哪两个因素最关键

风格参考：01_led 第 5 章

### 第 6 章：先认识本课里出现的核心名词

目的：在遇到代码之前，先把所有"陌生词"讲一遍。

必须包含：

- 6.1, 6.2, 6.3...等小节，每一小节讲一个名词
- 小节数量不能为了省事压缩；凡是本课第一次出现、学习者可能不懂的词，都要单独成节
- 一节只讲一个名词，不要把多个寄存器/API 混在一个项目符号里
- 每个名词都用这个模板讲：

  ```
  ### 6.X XXXX 是什么

  XXX 全称是：
  - （英文全称）

  中文通常叫：
  - （中文名）

  它的作用是：
  - （作用1）
  - （作用2）

  你可以先把它理解成：
  - （通俗比喻或简洁总结）

  在本课里...
  ```
- 必须涵盖：RCC/GPIO/寄存器/位掩码宏/HAL API/回调函数等所有新概念
- 不能有"你先记住"这样的替代品
- 不能写成项目符号清单后就结束，必须像 `01_led` 第 6 章那样展开解释
- 每个名词还必须补充：
  - 它属于哪个硬件/软件层
  - 它控制哪种行为
  - 本课为什么正好需要它
  - 写错、漏写、顺序错会出现什么具体现象

必须避免的粗糙写法：

```text
- `XXX`：XXX 是本课关键概念，负责某某链路。
```

合格写法应像 `01_led` 的 `RCC`、`RCC->APB2ENR`、`GPIOC->CRH` 那样，把名字、位置、作用、原因和现象都讲清楚。

风格参考：01_led 第 6 章

### 第 7 章：寄存器版代码逐步讲解

目的：让学习者知道底层到底在改什么。

必须包含：

- 7.1：先看完整逻辑（主函数逻辑）
- 7.2～7.N：逐个讲解每个函数和每个关键步骤

对于每个讲解，必须说明：

- 这一步在做什么
- 为什么需要这一步
- 为什么要这样改这个寄存器（而不是别的）
- 这一步改了哪几 bit
- 为什么要先清零再设置
- 如果忘记这一步会怎样
- 当前语句依赖前面哪一步已经完成
- 当前语句执行后，硬件状态发生了什么变化
- 这一步与第 5 章脑图中的哪一步对应

必须有足够小节支撑逐步讲解。例如一个定时器课程至少应拆出：

- `main()` 总流程
- 系统时钟为什么先配置
- GPIO 引脚为什么这样配
- RCC 外设时钟为什么要打开
- PSC/ARR/CCR 等关键寄存器的计算来源
- CCMR/CCER/DIER/SR/CR1 等控制位的含义
- 中断函数或主循环如何处理结果

不允许只写一段概述后放几行代码。第 7 章要让学习者即使不懂寄存器，也能顺着文档逐行读懂 `reg/src/main.c`。

风格参考：01_led 第 7 章

### 第 8 章：HAL 版代码逐步讲解

目的：让学习者知道 HAL 是怎么为你做这些事的。

必须包含：

- 8.1：HAL 版和寄存器版的本质差异
- 8.2～8.N：逐个讲解每个 HAL API 和参数

对于每个讲解，必须说明：

- 这个 API 是干什么的
- 它的参数是什么
- 参数代表什么硬件配置
- 这个 API 在底层等价于寄存器版的哪一步
- 如果这个 API 调用顺序变了会怎样
- 结构体为什么要先清零
- 每个字段的值为什么这样选
- HAL 的回调函数由谁调用，不是用户自己随便调用
- HAL 版哪些步骤对应寄存器版的“开时钟、配模式、开中断、清标志、启动外设”

必须避免：

- 只写“HAL 不是魔法”
- 只列 API 名字
- 只说“这个 API 完成初始化”
- 不解释结构体字段

合格目标是：学习者看完第 8 章，能把 HAL 每个字段反推回寄存器版的配置意图。

风格参考：01_led 第 8 章

### 第 9 章：两个版本真正应该怎么学

目的：告诉学习者这两个版本的学习策略和关系。

必须包含：

- 9.1：为什么先学寄存器版
- 9.2：为什么再看 HAL版
- 9.3：正确的心智模型（HAL API <-> 寄存器操作的映射）

风格参考：01_led 第 9 章

### 第 10 章：检验问题清单

**这一章是必须的，不可省略。**

目的：让学习者自我验证是否真的理解了。

必须包含：

- 10.1, 10.2...10.N：6~10 个核心问题，每个都有标准答案
- 问题类型必须包括：
  - 概念理解型：为什么 XXXX 是这个样子
  - 对比型：XXX 和 YYY 有什么区别
  - 排错型：如果出现某现象，应该先查什么
  - 原因型：为什么顺序不能乱
  - 反向型：如果忘记某步会怎样
- 答案要简洁但完整，关键知识点不能省略
- 每个答案用"**答**："开头
- 问题必须来自本课真实代码和真实现象，不能使用跨课程通用问题模板
- 至少 2 个问题要问“如果忘了某一步会怎样”
- 至少 2 个问题要问“为什么这个值这样算”
- 至少 1 个问题要问 HAL 和寄存器版的对应关系
- 至少 1 个问题要问具体排错顺序

参考格式：

```
1. **问题描述？** 补充说明细节？
   - **答**：简洁回答，但关键点不省略。

2. **另一个问题？**
   - **答**：...
```

风格参考：01_led 第 10 章

### 第 11 章：工程实现步骤

**这一章是必须的，不可省略。**

目的：让学习者知道如何从"学习"过渡到"工程应用"。

必须包含：

- 11.1：需求分析

  - 这个 Demo 的需求是什么
  - 需要用哪个硬件、哪个外设、哪些功能
- 11.2：硬件核查

  - 需要验证什么硬件条件
  - 如何查数据手册确认特性
- 11.3：寄存器实现路线

  - 按正确顺序列出每一大步（通常 3~8 步）
  - 每步说明为什么不能省略、为什么顺序不能乱
  - 每步说明这步如果出错会导致什么
- 11.4：HAL 实现路线

  - 按正确顺序列出每一大步
  - 说明和寄存器路线的对应关系
  - 说明为什么这样调用才能保证程序工作
- 11.5：工程思维

  - 学习阶段：为什么先学寄存器版
  - 工程阶段：为什么用 HAL版
  - 长期维护：应该关注什么
- 11.6：常见工程陷阱

  - 列举 3~5 个本课最常见的真实错误
  - 每个陷阱说明：错误现象、错误原因、正确做法

本章不能写成：

```text
1. 进入 reg 执行 pio run
2. 进入 hal 执行 pio run
```

编译下载步骤可以出现，但不能替代工程实现步骤。工程实现步骤讲的是“真实项目里怎么设计这条链路”，不是“命令怎么敲”。

参考格式：

```

- 需要的硬件是什么

### 11.2 硬件核查

1. 第一大步：说明为什么
2. 第二大步：说明为什么
...

### 11.4 HAL 实现路线

按这个顺序：

1. 第一大步：对应寄存器的哪一部分
2. 第二大步：...
...

### 11.5 工程思维

...

### 11.6 常见工程陷阱
```
