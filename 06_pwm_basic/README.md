# 第 6 课：PWM 基础

## 1. 本课到底在学什么

上一课你已经学会了：

- 定时器怎么计数
- 怎么通过 `PSC + ARR` 定义一个周期
- 怎么在周期结束时触发更新中断

这节课我们继续往前走一步：

- 不再只是“到点触发一次中断”
- 而是让定时器在引脚上直接输出一个周期性波形

这个波形就是：

- PWM

本课表面上的现象是：

- 一个外接到 `PA0` 的 LED 亮度发生变化

但真正要学的是：

- PWM 的频率是什么
- PWM 的占空比是什么
- 为什么占空比会影响 LED 亮度
- 定时器的 `CCR` 为什么能决定高电平持续多久
- 为什么定时器输出比较模式能生成 PWM
- 为什么做 PWM 时 GPIO 不能再是普通输出，而要切到复用功能输出

## 5. 先建立一个最基本的脑图

本课整条链路如下：

1. 先把系统时钟配置到 72MHz
2. 打开 GPIOA 和 TIM2 的时钟
3. 把 `PA0` 配置成“复用推挽输出”
4. 配置 `TIM2` 的基础计数频率和周期
5. 配置 `TIM2_CH1` 工作在 PWM 模式
6. 设置 `CCR1`
7. 启动定时器和通道输出
8. 引脚 `PA0` 自动输出 PWM 波形
9. LED 因占空比不同而表现出不同亮度

最关键的理解点是：

- PWM 不是 CPU 一直在手动拉高拉低引脚
- 而是定时器硬件按照配置自动输出波形

## 6. 先认识本课里出现的核心名词

### 6.1 PWM 是什么

PWM 全称是：

- Pulse Width Modulation

中文一般叫：

- 脉宽调制

它本质上是一种：

- 周期固定
- 高低电平反复切换
- 但高电平持续时间可调

的方波。

### 6.2 什么叫 PWM 频率

PWM 频率就是：

- 这个波形每秒重复多少次

例如：

- 1kHz

表示：

- 每秒重复 1000 个周期

### 6.3 什么叫占空比

占空比就是：

- 一个周期里，高电平所占的比例

例如：

- 25%：一个周期中 25% 时间高电平，75% 时间低电平
- 50%：高低各占一半
- 75%：大部分时间高电平

### 6.4 为什么占空比会影响 LED 亮度

因为 LED 的亮灭速度如果足够快，人眼会看到一个平均亮度。

所以：

- 高电平时间越长
- LED 平均导通时间越长
- 看起来就越亮

这就是 PWM 调光的核心原理。

### 6.5 `TIM2_CH1` 是什么

它表示：

- `TIM2` 的通道 1

通道可以理解成：

- 这个定时器对外输出或比较的一个独立功能口

在本课中：

- `TIM2_CH1` 会映射到 `PA0`

### 6.6 `CCR1` 是什么

`CCR1` 是：

- Capture/Compare Register 1

在 PWM 模式里，你可以先把它理解成：

- 决定“高电平持续到什么时候”

例如：

- `ARR = 999`
- `CCR1 = 250`

就意味着：

- 一个周期总共有 1000 个计数
- 前 250 个计数保持高电平
- 后 750 个计数保持低电平

所以占空比约为：

- 25%

### 6.7 `ARR` 和 `CCR1` 在 PWM 里分别管什么

你可以先这样理解：

- `ARR` 决定一个周期总长度
- `CCR1` 决定高电平长度

于是：

- PWM 频率主要由 `PSC + ARR` 决定
- PWM 占空比主要由 `CCR1 / (ARR + 1)` 决定

### 6.8 为什么 PWM 输出脚要配置成复用推挽输出

因为这节课的引脚电平不再由普通 GPIO 输出寄存器直接控制，而是：

- 交给定时器通道硬件控制

所以 GPIO 必须切换成：

- Alternate Function Output

在 F103 上常见写法就是：

- 复用推挽输出

### 6.9 `CCMR1` 是什么

`CCMR1` 是：

- Capture/Compare Mode Register 1

本课里它负责定义：

- 通道 1 处于什么输出模式

比如：

- PWM mode 1

### 6.10 `CCER` 是什么

`CCER` 是：

- Capture/Compare Enable Register

本课里它负责：

- 打开通道输出

### 6.11 `EGR` 是什么

`EGR` 是：

- Event Generation Register

本课里你会看到：

- 用它手动产生一次更新事件

它的作用是：

- 让刚写进去的预分频和重装配置立即生效

### 6.12 PWM mode 1 可以怎么理解

对当前阶段，你可以先粗略理解成：

- 当计数器 `CNT` 小于 `CCR1` 时输出高电平
- 当计数器 `CNT` 大于等于 `CCR1` 时输出低电平

这就自然形成了一个占空比可控的周期波形。

## 7. 本课 Demo 要实现什么

本课会让：

- `TIM2_CH1` 在 `PA0` 输出 1kHz PWM

参数设计如下：

1. 当前 `TIM2` 输入时钟按 `72MHz` 计算
2. 设 `PSC = 72 - 1`
3. 这样计数频率就是：
   - `72MHz / 72 = 1MHz`
4. 再设 `ARR = 1000 - 1`
5. 这样 PWM 频率就是：
   - `1MHz / 1000 = 1kHz`
6. 再设 `CCR1 = 250`
7. 占空比就是：
   - `250 / 1000 = 25%`

然后在主循环里逐步修改 `CCR1`，让 LED 亮度变化。

- 50%
- 90%

## 9. HAL版代码逐步讲解

对应代码文件：

- [hal/src/main.c](/home/myself/workspace/mcu/stm32/STM32F103C8T6/06_pwm_basic/hal/src/main.c)

### 9.1 HAL 版里为什么除了 `TIM_HandleTypeDef` 还会有 `TIM_OC_InitTypeDef`

因为 PWM 配置分成两层：

1. 定时器基础计数怎么配
2. 某个通道输出模式怎么配

所以：

- `TIM_HandleTypeDef` 负责定时器整体
- `TIM_OC_InitTypeDef` 负责通道输出比较/PWM 细节

### 9.2 `HAL_TIM_PWM_Init()` 是什么

它用于初始化定时器的 PWM 工作基础。

你可以把它理解成：

- 在基础定时器的基础上，进一步准备 PWM 所需的输出模式框架

### 9.3 `HAL_TIM_PWM_ConfigChannel()` 是什么

它负责：

- 指定某个通道输出什么模式
- 初始脉冲值是多少
- 极性是什么

也就是：

- 把 PWM 相关的“通道级配置”写进去

### 9.4 `HAL_TIM_PWM_Start()` 是什么

它负责：

- 真正启动某个 PWM 通道的输出

如果只初始化不启动，通道不会真正往引脚输出波形。

## 10. 寄存器版和 HAL版对比

### 10.1 寄存器版重点看什么

- 为什么 `PA0` 要切到复用推挽输出
- `PSC / ARR / CCR1` 的分工
- `CCMR1 / CCER / EGR / CR1` 分别在 PWM 链路中的作用

### 10.2 HAL版重点看什么

- HAL 如何把 PWM 拆成“定时器配置 + 通道配置 + 启动”
- `Pulse` 字段如何对应到底层 `CCR1`

### 10.3 两者如何一一对应

- 寄存器版写 `PSC / ARR` <-> HAL 的 `htim2.Init.Prescaler / Period`
- 寄存器版写 `CCR1` <-> HAL 的 `sConfigOC.Pulse`
- 寄存器版写 `CCMR1` 的 PWM 模式 <-> HAL 的 `OCMode = TIM_OCMODE_PWM1`
- 寄存器版写 `CCER` 开输出 <-> HAL 的 `HAL_TIM_PWM_Start()`

## 12. 常见问题排查

### 12.1 LED 完全不亮

优先检查：

1. 外接 LED 和电阻接线是否正确
2. 是否真的接在 `PA0`
3. `PA0` 是否配置成复用推挽输出
4. `TIM2` 时钟是否打开
5. PWM 通道是否真正启动

### 12.2 LED 一直常亮或常灭

优先检查：

1. `CCR1` 是否一直等于 0 或接近 `ARR`
2. 通道模式是否真的是 PWM mode 1
3. 引脚是否还停留在普通 GPIO 输出模式

### 12.3 频率不对

优先检查：

1. `TIM2` 输入时钟是否按 72MHz 计算
2. `PSC / ARR` 是否算错
3. 是否忘了 APB1 定时器倍频规则

### 12.4 HAL 版“亮一下就卡死”

如果你把本课 HAL 示例迁到一个“只有 `main.c` 的最小 PlatformIO 工程”里，
很容易遇到这种现象：

- 程序一开始像是跑了一下
- 然后 LED 就不再变化
- 如果你在调试器里停下，可能发现 CPU 卡在默认中断处理函数里

这通常不是 PWM 本身的问题，而是：

- 你调用了 `HAL_Init()`
- 你又调用了 `HAL_Delay()`
- 但工程里没有实现 `SysTick_Handler()`

为什么会这样：

1. `HAL_Init()` 会配置 SysTick
2. `HAL_Delay()` 依赖 SysTick 每 1ms 递增一次 tick
3. 如果没有 `SysTick_Handler()`，一旦 SysTick 中断到来
4. CPU 就会掉进启动文件里的默认弱符号死循环

所以在最小 HAL 工程里，至少要补上：

```c
void SysTick_Handler(void)
{
    HAL_IncTick();
}
```

这也是为什么“同样是 F103”：

- 有些 C8T6 示例工程看起来直接就能跑
- 但你自己在 PlatformIO 新建的最小 RCT6 工程却会卡

真正差异往往不是芯片型号，而是：

- 前者已经自带完整工程骨架
- 后者只有 `main.c`，缺少中断支撑文件

### 12.5 想把 PWM 改到别的引脚，结果 LED 常亮

这是很多人第一次从 `PA0/TIM2_CH1` 改到别的 LED 脚时最容易踩的坑。

先记住一个大前提：

- 不是所有 GPIO 都带定时器 PWM 功能

例如本课的 `PA0` 能做 PWM，是因为它本身就是：

- `TIM2_CH1`

但如果你改到别的脚，比如 `PA10`、`PA12`，就必须先确认：

1. 这个引脚是不是某个定时器通道
2. 它属于普通定时器还是高级定时器
3. 板上 LED 是高电平点亮还是低电平点亮

### 12.6 为什么改成 `PA10/TIM1_CH3` 后会常亮

这是一个非常典型的“定时器已经配了，但输出没真正出来”的案例。

如果你把本课代码从 `TIM2_CH1/PA0` 改到 `TIM1_CH3/PA10`，常见漏项有这几个：

1. **没有打开通道输出使能 `CC3E`**

   只配置 `CCMR2` 里的 PWM 模式还不够，必须再让通道真正对外输出：

   ```c
   TIM1->CCER |= TIM_CCER_CC3E;
   ```

2. **忘了 TIM1 还需要主输出使能 `MOE`**

   `TIM1` 是高级定时器，不像 `TIM2` 那样只开 `CEN` 就够。
   还必须：

   ```c
   TIM1->BDTR |= TIM_BDTR_MOE;
   ```

3. **LED 极性和 PWM 极性没对上**

   如果板上 LED 是“低电平点亮”，那你不能直接套“高电平点亮”的默认理解。
   要么改输出极性，要么改 PWM 模式，要么从占空比解释上整体反过来理解。

4. **占空比变量类型不够大**

   如果你把占空比设计成 `0..1000`，却写成：

   ```c
   uint8_t duty_cycle;
   ```

   那它最大只能到 `255`，后面的 `>1000` 检查也永远不会成立。

5. **步进变量写成无符号数，却又给它赋负值**

   例如：

   ```c
   uint32_t step;
   step = -1;
   ```

   这不会得到“递减 1”，而会下溢成一个很大的正数。
   这种 bug 很容易让亮度变化逻辑直接失控。

### 12.7 TIM2 和 TIM1 的思维差异

本课默认使用 `TIM2`，它是一个普通定时器，入门时更顺手。

你可以先这样记：

- `TIM2`：普通定时器，做基本 PWM 更直观
- `TIM1`：高级定时器，功能更强，但配置链更长

对初学者最重要的现实区别是：

- `TIM2` 通常配置完 `PSC/ARR/CCRx/模式/CCxE/CEN` 就够了
- `TIM1` 除了这些，还常常要记得 `BDTR.MOE`

所以第一次学 PWM，优先从：

- `PA0 -> TIM2_CH1`

开始，是最省脑子的路径

### 12.8 从一个完整周期去理解“为什么会亮、为什么会常亮”

很多人虽然会背：

- `PSC` 分频
- `ARR` 定周期
- `CCR` 定占空比

但一改引脚、一改极性，脑子里就乱了。  
所以这里把一个 PWM 周期真正拆开说一次。

假设：

- 定时器向上计数
- `ARR = 999`
- `CCR = 250`
- 工作在 `PWM mode 1`

那么一个周期里会发生什么？

1. `CNT` 从 `0` 开始往上数
2. 当 `CNT < CCR` 时，通道输出处于“有效电平”
3. 当 `CNT >= CCR` 时，通道输出处于“无效电平”
4. 当 `CNT` 数到 `ARR` 后回到 `0`，开始下一个周期

这就意味着：

- 一个周期总共有 `1000` 个计数
- 前 `250` 个计数输出有效电平
- 后 `750` 个计数输出无效电平

所以占空比约为：

- `250 / 1000 = 25%`

关键点在这里：

- `PWM mode 1` 只是在说“什么时候输出有效电平”
- 但“有效电平到底是高还是低”，还要看输出极性

也就是说，真正决定 LED 亮灭的是两层：

1. **比较逻辑**
   - 什么时候输出“有效”
   - 什么时候输出“无效”
2. **输出极性**
   - “有效”到底对应高电平还是低电平

如果 LED 是：

- 高电平点亮

那通常你会希望：

- 有效电平 = 高电平

如果 LED 是：

- 低电平点亮

那通常你会希望：

- 有效电平 = 低电平

这就是为什么：

- 同样一个 `CCR = 250`
- 换一块板、换一个 LED 接法
- 现象可能完全反过来

你看到“常亮”时，不一定是 PWM 没跑，也可能是：

1. 通道根本没输出，GPIO 停在一个固定电平
2. PWM 在输出，但极性和 LED 接法正好反着
3. `CCR` 已经接近 `ARR`，导致大部分时间都处于点亮状态

所以以后排 PWM 故障，建议按这个顺序想：

1. 这个脚是不是真的有定时器通道功能
2. 通道输出使能 `CCxE` 开了没有
3. 如果是 `TIM1/TIM8`，`MOE` 开了没有
4. 这是 `PWM mode 1` 还是别的模式
5. 当前“有效电平”到底是高还是低
6. 这颗 LED 到底是高电平点亮还是低电平点亮
7. `CCR` 当前到底设成了多少

## 13. 本课要点总结

本课最核心的结论如下：

1. PWM 是周期固定、占空比可调的波形
2. 占空比决定 LED 平均亮度
3. `ARR` 定周期，`CCR1` 定高电平宽度
4. 定时器硬件可以自动输出 PWM，不需要 CPU 手动翻转引脚
5. PWM 输出脚必须切到复用功能输出
6. HAL 的 PWM 配置本质上仍然围绕 `PSC / ARR / CCR / 通道模式`

如果你是沿着本仓库的 C8T6 示例学寄存器/HAL，然后自己在别的目录里
新建一个最小 `platformio.ini + main.c` 工程，下面这几点很重要：

1. 这不等于你复制了一份完整示例工程
2. HAL 示例往往依赖“主函数之外”的基础支撑
3. 最常见缺失就是：
   - `SysTick_Handler()`
   - `HAL_MspInit()` 或最小 MSP 初始化
   - 某些中断向量处理函数

所以你以后如果发现：

- 寄存器版能跑
- HAL 版编译也过了
- 但运行时像是突然卡死

优先不要怀疑芯片兼容性，而要先怀疑：

- 你的最小工程是不是把 HAL 所依赖的骨架文件丢了
