# 第 42 课：FSMC SRAM 教学模拟

## 1. 本课到底在学什么

本课表面现象是：程序向 `g_fake_sram[256]` 写入一组固定数据，再读回来校验；如果没有错误，PC13 LED 周期翻转。

真正要学的是外部 SRAM 的最小读写思想，但当前 BluePill 工程没有真实 FSMC 引脚，也没有配置 FSMC 控制器。源码用一个 `volatile uint16_t` 数组模拟“外部 SRAM 存储区”，让你先把“写入 -> 读回 -> 校验 -> 错误计数 -> LED 反馈”这条软件验证链路跑通。

这节课的位置很重要：它不是说 BluePill 已经能接 FSMC SRAM，而是给后面真正 FSMC 板子铺路。真实 FSMC 课会把数组地址换成外部存储器映射地址，把普通数组读写换成 FSMC 总线上的地址线、数据线、片选、读写时序。

## 2. 本课学习目标

学完本课，你应该能回答：

1. 当前源码为什么叫“FSMC SRAM 教学模拟”，而不是真实 FSMC SRAM 驱动？
2. `g_fake_sram[256]` 在本课里模拟了外部 SRAM 的哪一部分行为？
3. 为什么数组要声明成 `volatile`？
4. `0xA500 | i` 和 `0x5500 | i` 这种测试图案有什么作用？
5. `g_sram_errors` 什么时候增加，LED 为什么只在错误为 0 时翻转？
6. 真实 FSMC SRAM 相比当前模拟，还需要哪些硬件信号？
7. HAL 版和寄存器版在本课里真正不同的是哪一层？
8. 如果 LED 不闪，如何区分是 GPIO 问题还是 SRAM 校验问题？

## 4. 实验硬件

- 开发板：STM32F103C8T6 BluePill
- 下载器：ST-Link
- LED：PC13
- 真实外部 SRAM：本课不接
- 真实 FSMC 总线：BluePill/F103C8 不具备完整可用 FSMC 引脚

因此，本课上板观察的是 PC13 和调试变量，不是外部 SRAM 芯片波形。真实 FSMC SRAM 需要带 FSMC 引脚的 F103V/F103Z 等封装和对应外部 SRAM 电路。

## 5. 先建立一个最基本的脑图

```text
当前可编译 Demo
  -> 配 72MHz 系统时钟
  -> 配 PC13 输出
  -> g_fake_sram 作为模拟 SRAM 区
  -> for 循环写入 256 个半字
  -> for 循环读回 256 个半字
  -> 不匹配则 g_sram_errors++
  -> 主循环中 errors==0 才翻转 PC13

真实 FSMC SRAM 将来会变成
  -> 配 FSMC GPIO 复用推挽
  -> 配 FSMC Bank 控制寄存器
  -> 配 FSMC 时序寄存器
  -> 访问外部 SRAM 映射地址
  -> 用同样的写入/读回/校验方法验证硬件
```

当前代码重点训练的是校验方法和存储访问思路。不要把 `g_fake_sram` 误认为真实 FSMC 寄存器配置。

## 6. 先认识本课里出现的核心名词

### 6.1 `FSMC` 是什么

FSMC 是 Flexible Static Memory Controller，灵活静态存储控制器。

它是 STM32 某些型号内部的外部存储器控制器，可以把 SRAM、NOR Flash、8080 LCD 这类并行设备映射到 CPU 地址空间。当前 F103C8 BluePill 工程没有真正启用 FSMC，本课只讲概念边界和验证思路。

### 6.2 `外部 SRAM` 是什么

外部 SRAM 是接在 MCU 外部总线上的静态 RAM 芯片。

真实硬件里，CPU 对某个外部映射地址读写，FSMC 会自动输出地址线、数据线、片选和读写控制信号。本课用普通数组模拟“可读可写的半字存储区”。

### 6.3 `g_fake_sram[256]` 是什么

`g_fake_sram` 是当前源码里的模拟 SRAM 缓冲区。

它属于 C/RAM 软件层，不属于 FSMC 外设，也不在外部总线上。它让本课在 BluePill 上可编译、可运行、可调试，先验证读写校验流程。

### 6.4 `volatile uint16_t` 是什么

`uint16_t` 表示每个元素是 16 位半字，贴近常见 FSMC SRAM 的 16 位数据宽度。

`volatile` 告诉编译器不要把读写优化掉。本课需要真实执行“写数组、再读数组”的动作，方便调试器观察，也避免优化器因为结果看似可推导而简化校验过程。

### 6.5 `0xA500 | i` 是什么

这是寄存器版使用的测试图案。

高字节固定为 `0xA5`，低位包含索引 `i`，这样每个位置的数据都不同。若地址索引错、读回顺序错、某个元素没写进去，校验更容易发现。

### 6.6 `0x5500 | i` 是什么

这是 HAL 版使用的测试图案。

它和寄存器版的 `0xA500 | i` 作用相同，只是固定高位不同，便于你在调试器里区分当前下载的是 HAL 版还是寄存器版。

### 6.7 `g_sram_errors` 是什么

`g_sram_errors` 是读回校验错误计数。

写入后再次读取，如果某个元素不等于预期值，就自增。主循环只在 `g_sram_errors == 0` 时翻转 LED，所以 LED 不闪时要同时看 GPIO 和这个变量。

### 6.8 `PC13` 在本课做什么

PC13 是校验结果反馈。

它不是 SRAM 总线的一部分。当前 Demo 没有外部 SRAM 波形，所以用 PC13 表示“模拟读写校验通过后主循环仍在运行”。

### 6.9 `FSMC_BCR / FSMC_BTR` 是什么

它们是真实 FSMC SRAM 会用到的控制和时序寄存器。

当前源码没有访问这些寄存器。这里提到它们是为了建立后续真实 FSMC 的入口：`BCR` 控制存储器类型、数据宽度、Bank 使能；`BTR` 控制地址建立时间和数据保持时间。

### 6.10 `NE / NOE / NWE` 是什么

它们是真实 FSMC 总线上的控制信号。

`NE` 是片选，`NOE` 是读使能，`NWE` 是写使能。当前 BluePill 模拟代码没有这些引脚波形；真实 SRAM 接线错误时，常见现象是读回全 `0xFFFF`、全 `0x0000` 或随机值。

### 6.11 `HAL_SRAM_Init()` 是什么

`HAL_SRAM_Init()` 是 HAL 里用于真实外部 SRAM 的初始化 API。

当前源码没有调用它。真实 FSMC 工程中，它会根据 SRAM 句柄和时序结构体配置 FSMC 控制器。本课只把它作为后续知识入口，不把它写成已执行代码。

### 6.12 `内部 SRAM` 和 `外部 SRAM` 有什么区别

内部 SRAM 在 STM32 芯片内部，CPU 通过内部总线直接访问，本课的 `g_fake_sram` 就在这里。

外部 SRAM 在芯片外部，需要 FSMC 这类控制器把 CPU 访问转换成引脚上的地址、数据、读写控制时序。当前代码只验证“存储区读写校验”这个软件方法，没有验证外部总线电气和时序。

### 6.13 `地址线错误` 在真实 SRAM 中是什么现象

真实外部 SRAM 如果某根地址线接错或时序不对，可能出现写 A 地址却影响 B 地址、相邻地址读回重复、某些固定地址总是错误等现象。

本课用 `0xA500 | i` 这种带索引的测试图案，就是为了给将来的地址线排错做准备。全写 `0xFFFF` 看起来更简单，但发现不了地址混叠。

### 6.14 `数据线错误` 在真实 SRAM 中是什么现象

真实外部 SRAM 如果某根数据线接错，读回值可能总有某一 bit 固定为 0 或 1，或者高低字节异常。

当前内部数组不会产生这种硬件错误，但你可以通过观察测试图案理解：预期 `0xA501` 读成 `0xA581`，就要怀疑某些数据位异常。

## 7. 寄存器版代码逐步讲解

### 7.1 `system_clock_72mhz_init()`

代码配置 Flash 等待周期、HSE、PLL x9、APB1 二分频，并切换到 PLL 作为系统时钟。

虽然本课只操作 RAM 数组，统一 72MHz 时钟能保持和前后课程一致，也让 LED 延时大致稳定。

### 7.2 `pc13_led_init()`

打开 GPIOC 时钟，配置 PC13 为推挽输出，并先输出高电平熄灭 LED。

如果 LED 不闪，先确认这部分没问题，再去看 `g_sram_errors`。不要一开始就怀疑“SRAM 模拟失败”。

### 7.3 `g_fake_sram` 的存储位置

`static volatile uint16_t g_fake_sram[256];` 分配在 MCU 内部 SRAM 中。

它不是外部 SRAM，也不需要开 FSMC 时钟。这个事实必须记住，否则会误以为本课已经完成真实 FSMC 配置。

### 7.4 写入测试图案

```c
for(uint16_t i=0;i<256U;i++)
    g_fake_sram[i]=(uint16_t)(0xA500U|i);
```

循环把 0 到 255 共 256 个位置写成不同值。真实外部 SRAM 调试也常用类似图案，因为它比全写同一个值更能发现地址线或索引错误。

### 7.5 读回校验

```c
for(uint16_t i=0;i<256U;i++)
    if(g_fake_sram[i]!=(uint16_t)(0xA500U|i))
        g_sram_errors++;
```

每个位置读回后和预期值比较。不一致就增加错误计数。当前数组在内部 SRAM，正常情况下错误应为 0。

这里要注意比较表达式里的两边都带 `i`。左边是从数组某个元素读出的实际值，右边是这个索引位置理论上应该保存的值。也就是说，校验不是只问“这个数组能不能读”，而是在问“第 i 个位置读出的数据是不是刚才写给第 i 个位置的数据”。这就是外部 SRAM 自检最基本的思想。

调试时建议在第二个 `for` 循环结束后停住，而不是停在循环中间。停在循环中间时，`g_sram_errors` 还没有完成统计，数组后面的位置也可能还没被完整检查。这个观察位置一错，很容易把正常的中间状态误判成校验失败。

### 7.6 为什么不用中断

本课读写校验是纯同步流程，不需要中断。

真实 FSMC SRAM 读写也通常表现为 CPU 访问存储器地址，不是靠中断搬运单个数据。后续若用 DMA 才会引入更多事件机制。

### 7.7 主循环判断错误数

```c
if (g_sram_errors == 0U) pc13_toggle();
```

只有校验通过才翻转 LED。若 `g_sram_errors` 非 0，LED 会停在当前状态，方便你知道校验失败。

这个设计把“错误检测”和“可见反馈”分成两层：`g_sram_errors` 是真实判断依据，LED 只是把判断结果变成肉眼现象。排错时不能只看 LED，因为 LED 不闪可能来自 GPIO、延时、程序没跑、断点停住，也可能来自校验失败。正确顺序是先看 `g_sram_errors`，再根据它决定往数组校验还是 PC13 输出方向查。

### 7.8 `delay_cycles()`

延时只用于让 LED 肉眼可见。

它不参与 SRAM 校验。调试时可以缩短延时或在主循环里设断点，直接观察 `g_sram_errors`。

### 7.9 为什么校验只执行一次

寄存器版在 `app_init()` 中完成写入和读回校验，主循环里只根据结果翻转 LED。

这样安排能把“初始化校验”和“运行反馈”分开。真实外部 SRAM 调试时也常先做一次启动自检，再决定是否进入后续应用。

### 7.10 如果想模拟持续写读

可以把写入和读回校验放进主循环，但那会改变本课含义：LED 不再只是显示一次自检结果，而是显示持续压力测试结果。

当前 README 按源码讲一次性校验，不把压力测试写成已经实现的功能。

### 7.11 为什么错误后 LED 不再翻转

主循环只有在 `g_sram_errors == 0` 时才调用 `pc13_toggle()`。

这是一种简单错误锁存：只要初始化校验出现过错误，错误计数非 0，LED 就停止周期翻转。真实项目里也常用类似方式阻止系统继续进入依赖存储器的业务。

如果你手动在调试器里把 `g_sram_errors` 改回 0，LED 会继续翻转。这说明主循环判断的是软件变量，不是重新自动修复了 SRAM。真实工程里一般不会靠手动改变量恢复，而是会重新初始化外设、重新测试存储器，或者进入错误处理分支。

### 7.12 从数组地址看当前实验边界

在 Watch 窗口里观察 `&g_fake_sram[0]`，它应该落在 STM32 内部 SRAM 区间，而不是外部存储器映射区间。

这条证据很重要：如果地址不是外部 Bank 地址，CPU 的读写就不会经过 FSMC 总线，也不会产生 NE、NOE、NWE 这些外部控制信号。当前课程借用了 SRAM 自检方法，但硬件路径仍然是 CPU 访问内部 SRAM。

### 7.13 为什么当前代码不会暴露真实总线问题

内部 SRAM 和 CPU 在同一颗芯片内部，地址线、数据线、片选线都不是你在板子上焊出来的外部连线。

所以当前代码能通过，不能证明真实外部 SRAM 接线正确。它只能证明三件事：循环边界正确、测试图案正确、读回比较逻辑正确。将来迁移到真实 FSMC 时，如果同样的校验失败，才开始进一步区分地址线、数据线、片选和时序问题。

## 8. HAL 版代码逐步讲解

### 8.1 `HAL_Init()`

HAL 版先初始化 HAL 基础环境和 SysTick。

本课 HAL 版后面使用 `HAL_Delay(500)`，所以需要 `SysTick_Handler()` 调用 `HAL_IncTick()`。

### 8.2 HAL 时钟结构体

`RCC_OscInitTypeDef` 表达 HSE 和 PLL x9，`RCC_ClkInitTypeDef` 表达 SYSCLK、AHB、APB1、APB2 分频。

这些字段对应寄存器版的 RCC 配置，和 `g_fake_sram` 模拟读写没有直接关系。

### 8.3 `HAL_GPIO_Init()`

HAL 版用 `GPIO_InitTypeDef` 配置 PC13 输出。

`Pin=GPIO_PIN_13`，`Mode=GPIO_MODE_OUTPUT_PP`，`Speed=GPIO_SPEED_FREQ_LOW`，对应寄存器版 PC13 的 CRH MODE/CNF 设置。

### 8.4 HAL 版测试图案

HAL 版写入：

```c
g_fake_sram[i]=(uint16_t)(0x5500U|i);
```

它和寄存器版的流程一致，只是图案不同。调试器里看到 `0x55xx`，就知道当前跑的是 HAL 版。

### 8.5 HAL 版校验错误计数

HAL 版同样读回每个元素，不一致就增加 `g_sram_errors`。

这部分没有 HAL SRAM API，因为当前源码没有真实外部 SRAM，也没有 FSMC 句柄。

HAL 版的校验逻辑仍然是 C 语言数组逻辑。`HAL_Init()`、`HAL_RCC_OscConfig()`、`HAL_GPIO_Init()` 都没有参与数组读写；它们只保证系统基础环境和 LED 反馈能工作。把这几层分开后，看到 HAL 版没有 `HAL_SRAM_Init()` 就不会慌，因为源码本来没有访问外部 SRAM。

### 8.6 `HAL_GPIO_TogglePin()`

校验通过时，主循环调用 `HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13)` 翻转 LED。

它对应寄存器版 `pc13_toggle()`，只是底层读写 ODR/BSRR 的动作由 HAL 封装。

### 8.7 `HAL_Delay(500)`

HAL 延时让 LED 翻转周期可见。

如果 HAL 版卡住不闪，除了看 `g_sram_errors`，还要看 SysTick 是否正常进入，因为 `HAL_Delay()` 依赖 tick。

### 8.8 为什么没有 `HAL_SRAM_Init()`

因为当前工程不是完整 FSMC SRAM 硬件实验。

如果将来换到带 FSMC 的板子，HAL 版才会增加 SRAM 句柄、FSMC 时序结构体和 `HAL_SRAM_Init()`；当前 README 不把未出现的 API 当作源码步骤。

### 8.9 `HAL_Delay()` 和寄存器版延时的区别

HAL 版用 `HAL_Delay(500)`，寄存器版用忙循环 `delay_cycles()`。

这只影响 LED 反馈节奏，不影响 SRAM 模拟校验。HAL 延时依赖 SysTick，寄存器版忙等待依赖 CPU 执行速度。

### 8.10 HAL 版为什么仍能直接用数组

HAL 是外设库，不改变 C 语言内存访问方式。

`g_fake_sram[i] = ...` 在 HAL 版和寄存器版里都是普通 C 数组写入。只有当目标地址换成 FSMC 外部映射地址时，硬件总线行为才会出现。

### 8.11 真实 HAL SRAM 工程会多哪些对象

真实工程通常会出现 `SRAM_HandleTypeDef`、`FSMC_NORSRAM_TimingTypeDef` 和 `HAL_SRAM_Init()`。

本课源码没有这些对象，所以当前第 8 章只讲 HAL GPIO/RCC 和数组校验。把未来对象写成当前步骤，会让学生误判代码已经配置了 FSMC。

### 8.12 从当前模拟迁移到真实 HAL SRAM 的第一处变化

第一处变化不是校验循环，而是“存储区地址来源”。

当前写 `g_fake_sram[i]`；真实 FSMC SRAM 通常会定义一个外部映射基地址，例如某个 Bank 对应的起始地址，然后通过指针访问该区域。校验循环仍可保留，但目标从内部数组变成外部总线地址。

### 8.13 第二处变化是时序配置

真实 SRAM 需要地址建立时间、数据建立时间、总线周转时间等时序参数。

这些参数不是为了“让代码更复杂”，而是为了匹配外部 SRAM 芯片的数据手册。时序太短会读写不稳定，时序太长会速度慢但通常更容易先跑通。

### 8.14 第三处变化是 GPIO 复用

FSMC 地址线、数据线和控制线都要配置成复用推挽输出或对应输入状态。

当前 BluePill 模拟没有这些引脚配置。真实板子上，如果 GPIO 复用没配好，FSMC 控制器内部配置正确也不会在引脚上产生有效总线波形。

### 8.15 真实 HAL 工程里错误处理会怎么放

真实 HAL SRAM 初始化通常会检查 `HAL_SRAM_Init()` 的返回值。返回不是 `HAL_OK` 时，程序应该进入错误处理，而不是继续访问外部地址。

当前 HAL 版没有这一步，因为没有调用 SRAM 初始化 API。它的错误处理点只有数组读回比较和 `g_sram_errors`。这也提醒你：HAL 返回值和数据校验不是同一个层次，前者说明初始化 API 是否成功执行，后者说明存储器读写结果是否可信。

### 8.16 真实 HAL 工程里校验循环还能不能复用

能复用，但要换访问目标。

当前目标是 `g_fake_sram[i]`。真实工程可以把目标换成指向外部 SRAM 映射区的 `volatile uint16_t *`，循环结构仍然是写图案、读回、计错。这样做的好处是教学路线连续：先证明校验算法，再把算法放到真实硬件地址上。

## 9. 两个版本真正应该怎么学

本课 reg/hal 的差别在 GPIO 和时钟初始化方式，不在 SRAM 模拟流程。两个版本的核心读写校验逻辑一样：写一组可识别数据，读回比较，错误计数，LED 反馈。

读这课时要分清三层：当前源码的内部数组模拟、真实 FSMC 的硬件总线概念、后续 HAL SRAM API 的工程封装。三者有关联，但不能混成同一件事。

## 10. 检验问题清单

### 10.1 当前代码有没有真正配置 FSMC？

**答**：没有。源码没有写 FSMC 寄存器，也没有调用 `HAL_SRAM_Init()`，只是用内部 RAM 数组模拟读写校验。

### 10.2 为什么 `g_fake_sram` 要用 `volatile`？

**答**：为了让写入和读回动作真实保留，方便调试观察，避免编译器把校验流程优化得不直观。

### 10.3 为什么测试图案要带索引 `i`？

**答**：每个地址写入不同数据，更容易发现地址、索引或读写顺序错误。

### 10.4 LED 不闪一定是 SRAM 错吗？

**答**：不一定。先看 PC13 GPIO 配置，再看 `g_sram_errors`。若错误数为 0 但 LED 不闪，多半是 GPIO 或延时问题。

### 10.5 真实 FSMC SRAM 还需要什么？

**答**：需要带 FSMC 的芯片封装、外部 SRAM 电路、地址/数据/控制线接线、FSMC GPIO 复用配置、Bank 控制和时序配置。

### 10.6 HAL 版为什么没有 SRAM 句柄？

**答**：因为当前 HAL 版也只是数组模拟。没有真实 FSMC 外设初始化，就不需要 SRAM 句柄。

### 10.7 `g_sram_errors` 正常应该是多少？

**答**：正常应为 0。内部数组读写没有外部总线不稳定因素，若非 0，通常是代码逻辑或调试观察位置的问题。

### 10.8 这课和真实 FSMC 的关系是什么？

**答**：它训练读写校验方法和总线概念边界。真实 FSMC 只是把存储区从内部数组换成外部映射地址，并增加硬件时序配置。

### 10.9 为什么不能把当前通过当成外部 SRAM 通过？

**答**：因为当前访问地址属于内部数组，没有经过 FSMC 引脚和外部芯片。它通过只能证明软件校验流程正确，不能证明外部地址线、数据线、片选线或时序正确。

### 10.10 观察 `&g_fake_sram[0]` 有什么意义？

**答**：它能证明变量位于哪段内存。若地址在内部 SRAM 区间，本课就是内部数组模拟；只有访问外部 Bank 映射地址，CPU 读写才可能经过 FSMC 总线。

## 11. 工程实现步骤

### 11.1 需求分析

先在 BluePill 上获得一个可编译、可观察的 SRAM 读写校验流程。

### 11.2 硬件核查

确认本课不需要外部 SRAM；只需要 PC13 LED 能反馈校验结果。

### 11.3 寄存器路线

配置时钟和 PC13，写入 `0xA500|i`，读回校验，错误为 0 时翻转 LED。

### 11.4 HAL 路线

用 HAL 配置时钟和 PC13，写入 `0x5500|i`，读回校验，错误为 0 时翻转 LED。

### 11.5 工程思维

做真实外部存储器前，先建立读写校验方法。否则硬件不通时，你分不清是接线、时序、地址还是验证方法本身错。

### 11.6 常见工程陷阱

把当前数组模拟误认为真实 FSMC、LED 不闪就直接怀疑 SRAM、全写同一个测试值导致地址问题不容易暴露、HAL 版卡在延时却误判为校验失败，都是本课要避免的误区。

## 12. 运行现象

正常情况下，程序写入并读回 256 个半字，`g_sram_errors` 保持 0，PC13 周期翻转。

你可以在调试器里观察 `g_fake_sram` 前几个元素：寄存器版应看到 `0xA500`、`0xA501`、`0xA502`；HAL 版应看到 `0x5500`、`0x5501`、`0x5502`。

如果在 `app_init()` 或 HAL 版第二个 for 循环后打断点，`g_sram_errors` 应该仍为 0。若你在第一个 for 循环中间停住，数组后半部分还没写完，这是正常的断点位置现象。

当前 Demo 即使没有任何外部 SRAM 接线也会通过，因为它测试的是内部数组。这个现象本身就是课程目标的一部分：先把软件校验方法和真实硬件访问分开。

## 13. 常见问题排查

### 13.1 PC13 不闪

先看 `g_sram_errors`。若为 0，检查 GPIOC 时钟、PC13 模式和 LED 硬件；若非 0，再查校验循环。

### 13.2 `g_sram_errors` 非 0

检查写入图案和读回比较是否一致，确认当前观察的是 reg 版还是 HAL 版，别把 `0xA5xx` 和 `0x55xx` 混用。

### 13.3 Watch 里数组看不到

确认变量名是 `g_fake_sram`，并确认调试器下载的是当前工程。数组是 `static` 文件内变量，有些调试器需要按符号选择查看。

### 13.4 HAL 版卡住

检查 `SysTick_Handler()` 是否调用 `HAL_IncTick()`，因为 `HAL_Delay(500)` 依赖 HAL tick。

### 13.5 接了真实 SRAM 但没反应

当前 BluePill 工程没有真实 FSMC 驱动。需要换带 FSMC 的芯片和工程，增加 FSMC GPIO、Bank、时序和外部映射地址访问。

### 13.6 校验一直通过但你以为在测外部 SRAM

确认 `g_fake_sram` 的地址。它是内部 RAM 全局数组，不是 `0x60000000` 这类外部存储器映射地址。当前通过只能证明内部数组读写逻辑没问题。

### 13.7 想迁移到真实 FSMC 时第一步查什么

先查芯片封装和开发板原理图是否真的引出了 FSMC 所需地址线、数据线、NE、NOE、NWE。没有硬件基础，软件配置再完整也不会得到真实 SRAM 访问。

### 13.8 真实迁移后只有偶发错误

如果换成真实 SRAM 后 `g_sram_errors` 偶尔增加，不要只改 C 循环。优先检查 FSMC 时序是否太激进、外部 SRAM 供电是否稳定、总线线长和焊接是否可靠。当前数组模拟不会出现这种偶发硬件问题，所以一旦迁移后出现随机错误，排查重点应转到硬件和时序层。

### 13.9 真实迁移后错误位置很规律

如果错误总出现在固定间隔的地址，例如每隔 2、4、8、16 个元素重复，通常要怀疑地址线映射、地址线短路或地址线未接好。如果读回值总是某一 bit 错，才更像数据线问题。这就是本课使用 `0xA500|i`、`0x5500|i` 的原因：图案带地址信息，后续才能根据错误形态反推硬件层问题。

## 14. 本课最核心的结论

1. 当前课程是 FSMC SRAM 的教学模拟，不是真实 FSMC 驱动。
2. `g_fake_sram` 位于内部 SRAM，只模拟可读写存储区。
3. 读写校验要使用可区分地址的测试图案，不能只写一个固定值。
4. `g_sram_errors` 是判断校验是否通过的核心变量。
5. reg/hal 的差异主要在时钟和 GPIO 初始化，校验逻辑相同。
6. 真实 FSMC 还需要硬件总线、GPIO 复用、Bank 控制和时序配置。
7. `&g_fake_sram[0]` 的地址能直接证明当前访问的是内部数组还是外部映射区。
8. 当前校验通过是迁移前的算法验证，不是外部 SRAM 硬件验收。

## 15. 建议你现在怎么读这节课

先确认这课不是在 BluePill 上驱动真实 FSMC。然后看 `app_init()` 或 HAL 版两个 for 循环，把写入图案、读回比较、错误计数和 LED 反馈连起来。

第二遍再带着真实 SRAM 的视角读：如果把 `g_fake_sram` 换成外部映射地址，哪些代码能保留，哪些地方必须增加 FSMC GPIO、Bank 和时序配置。这样迁移时不会推翻全部思路。
