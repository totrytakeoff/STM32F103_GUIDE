# 第 13 课：ADC + DMA 缓冲区

## 1. 本课要学什么

上一课 `12_dma_basic` 已经让你看到：

- `ADC1` 可以持续采样
- `DMA1_Channel1` 可以把 `ADC1->DR` 的值自动搬到 RAM

但上一课的数据目标只有一个变量：

```c
g_adc_value
```

这只能保存"最新一次结果"。

这一课我们再往前走一步：

- 不只保存 1 个采样值
- 而是保存一整段采样缓冲区
- 然后由 CPU 对这段缓冲区做处理（平均滤波）

所以这节课真正要学的是：

1. 什么是 DMA 缓冲区
2. 为什么 DMA 要用"内存地址自增"（MINC=1）
3. 为什么 `CNDTR` 不再是 1，而是一个数组长度（16）
4. ADC + DMA 为什么常常配合"平均值滤波"
5. 平均值滤波为什么能让结果更稳定

### 本课与上一课的对比

| 对比项 | 12_dma_basic | 本课 13_adc_dma |
|--------|-------------|----------------|
| 目标地址 | `&g_adc_value`（变量） | `g_adc_buffer`（数组首地址） |
| CNDTR | 1 | 16 |
| MINC | 0（不自增） | **1（自增）** |
| 数据保留 | 只保留最新值 | 保留 16 个历史值 |
| 主循环处理 | 直接用一个值判断 | **对 16 个值求平均再判断** |
| 结果稳定性 | 单值可能抖动 | 平均值更平滑 |

## 2. 本课最终 Demo

### 2.1 硬件连接

- `PA1` 接 10k 电位器滑动端
- 电位器两端分别接 `3.3V` 和 `GND`
- `PC13` 使用 BluePill 板载 LED

### 2.2 运行现象

- `ADC1` 持续采样 `PA1`
- `DMA1_Channel1` 自动把采样结果依次写入一个长度为 16 的数组
- 主循环对这 16 个采样值求平均
- 平均值大于 `2048` 时点亮 `PC13`
- 平均值不大于 `2048` 时熄灭 `PC13`

你会看到的效果是：

- 与上一课相比，**LED 状态更稳定**
- 电位器在阈值附近来回晃动时，**抖动会更小**

这就是"采样缓冲区 + 简单平均滤波"的最小实际价值。

## 3. 本课数据流总图

这一课的数据流是：

```
PA1 模拟电压 → ADC1 持续转换 → ADC1->DR
    ↓（DMA 请求）
DMA1_Channel1（MINC=1, CNDTR=16, CIRC=1）
    ↓
g_adc_buffer[0] → [1] → ... → [15] → 回到 [0]（循环覆盖）
    ↓
CPU 遍历求和 → 求平均 → 与阈值比较 → 控制 LED
```

各部分的意思：

- `PA1`：模拟输入引脚
- `ADC1`：把模拟电压转换成 12 位数字值
- `ADC1->DR`：ADC 数据寄存器
- `DMA1_Channel1`：自动把 DR 的结果搬到数组里
- `g_adc_buffer[16]`：DMA 的目标缓冲区
- `求平均`：CPU 对缓冲区做简单处理
- `LED`：用来观察处理结果

---

## 4. 为什么这次要用数组

如果只保存一个结果，你很难做这些事：

- 平均滤波
- 最大值/最小值分析
- 峰值检测
- 波形采样缓存

而数组缓冲区可以让你：

- 一次性拿到一组采样点
- 再由 CPU 统一处理

这就是很多真实项目里 ADC + DMA 的基本套路。

## 5. DMA 循环缓冲区是如何工作的

### 5.1 图解时间线

以 CNDTR=4（简化版）为例：

```
时间 →   t0     t1     t2     t3     t4     t5     t6     t7
ADC 值： [100]  [200]  [300]  [400]  [500]  [600]  [700]  [800]
          ↓      ↓      ↓      ↓      ↓      ↓      ↓      ↓
buffer： [100]  [100]  [100]  [100]  [500]  [500]  [500]  [500]
         [  ]→ [200]  [200]  [200]  [200]→ [600]  [600]  [600]
         [  ]   [  ]→ [300]  [300]  [300]  [300]→ [700]  [700]
         [  ]   [  ]   [  ]→ [400]  [400]  [400]  [400]→ [800]
                    ↑ t0~t3：第一次填充
                              ↑ t4~t7：第二次循环覆盖
```

所以：

- DMA 先在数组里依序填充 16 个值
- 填满后因 CIRC=1，从头开始覆盖
- 任何时候数组里都保存着**最近 16 次采样值**
- 但这 16 个值的**时序顺序不是线性的**——你需要知道当前 DMA 写到了哪个位置

### 5.2 为什么仍然可以求平均

对于"求平均"这个操作，循环顺序不影响结果：

- 数组里始终是最近 16 次采样的结果
- 顺序不重要，总和重要
- 平均值 = 总和 / 16，与顺序无关

## 6. 本课寄存器版新增的关键点

### 6.1 缓冲区变量

我们不再使用单个变量，而是定义：

```c
static volatile uint16_t g_adc_buffer[16];
```

使用 `volatile` 的原因是：

- 这个数组会被 DMA 硬件异步修改
- 不是普通的"只由 C 代码顺序写入"的变量
- 如果不用 volatile，编译器可能把数组内容缓存在寄存器或 cache 中，导致读取到过时数据

### 6.2 `CNDTR = 16`（不再是 1）

```
DMA1_Channel1->CNDTR = ADC_BUFFER_SIZE;  // = 16
```

DMA 一轮传输的长度从 1 变成 16 个数据。

配合 CIRC=1：搬完 16 个后自动重载为 16，继续下一轮。

### 6.3 `MINC = 1`（内存地址自增）

这是本课最关键的新位。

MINC = Memory Increment

它的意思是：

- 每搬完一个数据后，内存地址自动指向下一个位置

如果不开 MINC：

- 16 次搬运都会写到同一个地址 g_adc_buffer[0]
- 后面的元素永远不被更新
- 数组完全失去意义

所以本课**必须打开 MINC**。

### 6.4 外设地址为什么不自增（PINC=0）

外设地址始终是 ADC1->DR，这是一个固定的寄存器地址。

每次转换结果都出现在同一个地址，所以 PINC=0。

### 6.5 循环缓冲区带来的实际问题

在循环模式下，CPU 读取数组的瞬间：

- DMA 可能正在写 buffer[5]
- CPU 读了 buffer[5] 但读到的是半新半旧的值？
  实际上 DMA 传输是原子的（一次传输一个半字），
  只要 g_adc_buffer 是 uint16_t，单个元素的读写是安全的。
- 但整个数组在多线程（DMA + CPU）环境下不是一致的

更健壮的方案（后续课程会涉及）：

1. **半传输中断**：DMA 搬完一半（8 个）触发中断，CPU 处理前一半
2. **双缓冲（乒乓缓冲）**：两个缓冲区轮流使用
3. **DMA 传输完成中断**：搬完一轮后通知 CPU

## 7. 本课寄存器版配置了一致的地方

### 7.1 GPIO

- `PC13` 配成输出
- `PA1` 配成模拟输入

### 7.2 DMA1_Channel1

这次重点配置：

| 寄存器 | 值 | 含义 |
|--------|-----|------|
| `CPAR` | `&ADC1->DR` | 外设源地址（固定） |
| `CMAR` | `&g_adc_buffer[0]` | 内存目标地址（数组首地址） |
| `CNDTR` | 16 | 每轮传输计数 = 数组长度 |
| `CCR.DIR` | 0 | 外设到内存 |
| `CCR.PINC` | 0 | 外设地址不自增 |
| `CCR.MINC` | **1** | **内存地址自增 ← 关键变化** |
| `CCR.PSIZE` | 01 (16bit) | 外设数据宽度 |
| `CCR.MSIZE` | 01 (16bit) | 内存数据宽度 |
| `CCR.CIRC` | 1 | 循环模式 |

### 7.3 ADC1

依然配置成：

- 单通道 `ADC1_IN1`
- 连续转换模式（CONT=1）
- ADC 结果触发 DMA（CR2.DMA=1）

## 8. 为什么主循环这次要"求平均"

如果你只是把数组搬过去，但主循环只看 `g_adc_buffer[0]`，那就没学到这节课真正的价值。

所以本课主循环会做：

1. 遍历 16 个采样值
2. 求和（用 uint32_t 避免溢出）
3. 计算平均值
4. 再用平均值判断 LED

这样你会真正感受到：

- **DMA 负责采集**
- **CPU 负责处理**

这正是很多嵌入式采样任务的分工方式。

### 8.1 求和为什么用 uint32_t

16 个 12 位 ADC 值的最大和 = 16 × 4095 = 65520。

而 uint16_t 的最大值是 65535，65520 已经很接近上限了。

如果改成 32 个采样值，uint16_t 直接溢出。所以用 uint32_t 安全。

## 9. HAL 版的核心变化是什么

HAL 版最大的变化不是 ADC，而是 DMA 配置中的一行：

```c
hdma_adc1.Init.MemInc = DMA_MINC_ENABLE;   // ★ 内存地址自增
```

以及调用时的参数：

```c
HAL_ADC_Start_DMA(&hadc1, (uint32_t *)g_adc_buffer, 16);
// 地址：数组首地址         ↑               ↑
// 长度：16                 ↑
```

### 9.1 与上一课 HAL 版的逐项对比

| 配置项 | 12_dma_basic | 本课 |
|--------|-------------|------|
| `MemInc` | `DMA_MINC_DISABLE` | **`DMA_MINC_ENABLE`** |
| `Start_DMA 地址` | `&g_adc_value` | `g_adc_buffer`（数组） |
| `Start_DMA 长度` | 1 | 16 |

就这三行变化，实现了从"单值搬运"到"缓冲区搬运"的升级。

## 10. 本课你要特别注意的几个点

### 10.1 `volatile`

缓冲区要声明为 `volatile`，原因：

- 这个数组被 DMA 硬件异步修改
- 不是普通的"只由 C 代码顺序写入"的变量

### 10.2 求和要用更大的整数类型

```c
uint32_t sum = 0;   // 不要用 uint16_t！
```

16 个 uint16_t 相加，结果可能超过 16 位范围。

### 10.3 DMA 与 CPU 会同时访问缓冲区

本课为了教学简单，直接在主循环里读。

这在最小 demo 里没问题，但你要知道：

- DMA 可能正在写数组
- CPU 也可能正在读数组
- 单个 uint16_t 的读取是原子的，但**整个数组不是** 

在更复杂的项目里，后面通常要引入：

- 半传输中断
- 全传输中断
- 双缓冲（乒乓缓存）
- RTOS 同步机制（信号量）

## 12. 本课容易出错的地方

### 12.1 忘了设 MINC=1

后果：16 个 ADC 值都写到 buffer[0]，其他元素始终为 0。

### 12.2 CNDTR 和数组长度不一致

如果 CNDTR=16 但数组只有 10 个元素，DMA 会写入数组越界。

### 12.3 数组没加 volatile

编译器可能优化掉对数组的读取，导致平均值永远不变。

### 12.4 求和用 uint16_t

16 × 4095 = 65520，非常接近 uint16_t 上限 65535，容易溢出。
