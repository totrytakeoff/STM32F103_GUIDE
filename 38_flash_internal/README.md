# 第 38 课：内部 Flash 读写

## 1. 本课到底在学什么

本课表面现象是：程序检查 STM32 内部 Flash 最后一页起始地址 `0x0800FC00` 是否已经保存 `0x1234`。如果没有，就擦除最后一页并写入半字 `0x1234`；之后 PC13 LED 根据读回结果快慢闪。

真正要学的是：内部 Flash 不是普通 RAM，不能随便改一个字节。写 Flash 必须遵守固定流程：

```text
解锁 Flash
  -> 等待 BSY 清零
  -> 页擦除
  -> 等待擦除完成
  -> 半字编程
  -> 等待编程完成
  -> 重新上锁
  -> 读回校验
```

这节课只写 64KB Flash 的最后一页第一个半字。它用于教学演示，不是完整参数存储库。真实工程中要考虑链接脚本、页内数据备份、磨损次数和掉电保护。

## 2. 本课学习目标

学完本课，你应该能回答：

1. 内部 Flash 和 SRAM 的读写方式有什么不同？
2. 为什么写 Flash 前必须先解锁？
3. `KEYR` 的两个钥匙值分别是什么？
4. 为什么写前要先页擦除？
5. `FLASH_LAST_PAGE = 0x0800FC00` 为什么是 64KB Flash 最后一页？
6. `PER`、`STRT`、`PG`、`LOCK` 分别控制什么？
7. `BSY` 为什么每个阶段都要等？
8. HAL 版 `HAL_FLASHEx_Erase()` 和 `HAL_FLASH_Program()` 对应哪些寄存器动作？

## 4. 实验硬件

- 开发板：STM32F103C8T6 BluePill
- 下载器：ST-Link
- LED：PC13

本课不需要外部存储器。写入对象是 MCU 片内 Flash。注意：本课会擦除最后一页 Flash，确保你的工程代码没有占用该页。

## 5. 先建立一个最基本的脑图

```text
Flash 地址空间
  -> 用户程序通常从 0x08000000 开始
  -> STM32F103C8 标称 64KB
  -> 最后一页起始地址 = 0x08000000 + 64KB - 1KB = 0x0800FC00

main()
  -> 读 0x0800FC00 的半字
  -> 如果不是 0x1234
     -> 解锁
     -> 擦除 0x0800FC00 所在页
     -> 写入半字 0x1234
     -> 上锁
  -> 循环读回
  -> 正确快闪，错误慢闪
```

Flash 写入最重要的直觉：只能把擦除后的单元按规则编程，擦除单位是页，不是一个变量。

## 6. 先认识本课里出现的核心名词

### 6.1 `内部 Flash` 是什么

内部 Flash 是 STM32 片内非易失存储器，程序代码就存放在这里。

它属于存储器映射层和芯片存储外设层。CPU 可以像读普通地址一样读它，但写入必须通过 Flash 控制器流程。

如果把 Flash 当 RAM 随便写，通常写不进去，甚至触发错误标志。

### 6.2 `0x08000000` 是什么

`0x08000000` 是 STM32F103 用户 Flash 常见起始地址。

程序下载后通常从这个区域取指执行。本课选择最后一页，是为了避开前面的代码区。

### 6.3 `FLASH_LAST_PAGE 0x0800FC00` 是什么

STM32F103C8 标称 Flash 64KB，页大小通常 1KB。

```text
0x08000000 + 0x10000 - 0x400 = 0x0800FC00
```

所以 `0x0800FC00` 是 64KB 空间最后 1KB 页的起始地址。

### 6.4 `页擦除` 是什么

Flash 擦除单位是页。本课擦除最后一页，整页内容都会变成擦除态。

这属于 Flash 硬件特性。即使只想改一个半字，也必须先考虑整页数据是否需要备份和恢复。

### 6.5 `半字编程` 是什么

STM32F1 Flash 常用半字，即 16 位写入。

本课写：

```c
*(__IO uint16_t*)FLASH_LAST_PAGE = 0x1234U;
```

它不是普通 RAM 赋值，而是在 `PG` 编程模式下触发 Flash 控制器编程。

### 6.6 `FLASH->KEYR` 是什么

`KEYR` 是 Flash 解锁钥匙寄存器。

写入两次固定钥匙：

- `0x45670123`
- `0xCDEF89AB`

才能清除锁定状态，允许擦除和编程。

### 6.7 `FLASH->SR.BSY` 是什么

`BSY` 是 Busy 忙标志。

擦除或编程期间 `BSY=1`。软件必须等待它清零，才能进行下一步，否则会在 Flash 仍忙时发起新操作。

### 6.8 `FLASH->CR.LOCK` 是什么

`LOCK` 是 Flash 控制寄存器里的锁定位。

上电后通常锁定。解锁后可以擦写；操作完成后重新置位 `LOCK`，防止程序误擦写 Flash。

### 6.9 `PER` 是什么

`PER` 是 Page Erase，页擦除使能位。

置位后，再设置擦除地址并写 `STRT`，Flash 控制器会擦除该页。

### 6.10 `AR` 是什么

`AR` 是 Address Register，擦除地址寄存器。

本课写入 `FLASH_LAST_PAGE`，告诉 Flash 控制器要擦除哪一页。

### 6.11 `STRT` 是什么

`STRT` 是 Start，启动擦除操作位。

在 `PER=1` 且 `AR` 设置好后，写 `STRT` 才真正开始页擦除。

### 6.12 `PG` 是什么

`PG` 是 Program，编程使能位。

置位后，对 Flash 地址执行半字写入会触发编程流程。写完后要清掉 `PG`。

### 6.13 `PGERR / WRPRTERR` 是什么

它们是 Flash 状态寄存器里的错误标志。

`PGERR` 常见于编程流程不符合要求，例如地址或写入方式不对；`WRPRTERR` 表示遇到写保护错误。寄存器版最小 Demo 只等待 `BSY`，但真实排错时要看这些错误位，否则会误以为“代码执行了就一定写成功”。

### 6.14 `HAL_FLASHEx_Erase()` 是什么

HAL 页擦除 API。通过 `FLASH_EraseInitTypeDef` 指定页地址和页数，底层完成等待、设置 PER/AR/STRT 和错误记录。

### 6.15 `HAL_FLASH_Program()` 是什么

HAL 编程 API。本课使用 `FLASH_TYPEPROGRAM_HALFWORD`，向 `FLASH_LAST_PAGE` 写入 `0x1234`。

## 7. 寄存器版代码逐步讲解

### 7.1 读取目标地址

```c
if(*(__IO uint16_t*)FLASH_LAST_PAGE!=0x1234U)
```

CPU 直接从 Flash 地址读取半字。读 Flash 不需要解锁。

### 7.2 `flash_unlock()`

如果 `FLASH->CR.LOCK` 为 1，依次写两个钥匙值到 `KEYR`。顺序错或值错，Flash 不会解锁。

### 7.3 `flash_wait()`

```c
while(FLASH->SR & FLASH_SR_BSY){}
```

等待 Flash 控制器空闲。擦除和编程前后都需要确认不忙。

### 7.4 进入页擦除模式

```c
FLASH->CR |= FLASH_CR_PER;
FLASH->AR = FLASH_LAST_PAGE;
FLASH->CR |= FLASH_CR_STRT;
```

`PER` 选择页擦除，`AR` 指定页地址，`STRT` 启动擦除。

### 7.5 等待擦除完成并退出擦除模式

擦除期间 `BSY=1`。完成后清 `PER`，避免后续误触发擦除流程。

### 7.6 进入编程模式

```c
FLASH->CR |= FLASH_CR_PG;
```

开启半字编程流程。此时对目标 Flash 地址写 16 位数据会触发硬件编程。

### 7.7 写入 `0x1234`

```c
*(__IO uint16_t*)FLASH_LAST_PAGE=0x1234U;
```

这句看起来像普通指针写，但在 `PG=1` 时由 Flash 控制器执行非易失写入。

### 7.8 等待编程完成并退出

等待 `BSY=0`，清 `PG`，再设置 `LOCK` 上锁。上锁是良好工程习惯。

### 7.9 错误标志应该怎么看

本课寄存器版代码为了保持主线清楚，没有展开 `PGERR` 和 `WRPRTERR` 的处理。调试时如果读回仍不是 `0x1234`，不要只看 `BSY`，还要观察 `FLASH->SR` 里的错误位。

如果错误位被置位，应先清错误标志，再重新按“擦除页 -> 编程半字 -> 读回校验”的顺序来判断。只重复写同一个地址，通常不会解决 Flash 流程错误。

### 7.10 主循环读回校验

循环读取目标半字。等于 `0x1234` 时快闪，否则慢闪。

## 8. HAL 版代码逐步讲解

### 8.1 `HAL_FLASH_Unlock()`

封装写 `KEYR` 两个钥匙值，解除 Flash 写保护。

### 8.2 `FLASH_EraseInitTypeDef`

结构体描述擦除类型：

- `TypeErase = FLASH_TYPEERASE_PAGES`：页擦除
- `PageAddress = FLASH_LAST_PAGE`：目标页
- `NbPages = 1`：擦 1 页

### 8.3 `HAL_FLASHEx_Erase()`

执行页擦除，底层设置 `PER/AR/STRT` 并等待完成。`err` 用于返回错误页信息。

如果它返回的不是 `HAL_OK`，或者 `err` 不是预期值，说明擦除阶段已经失败，后面的编程结果不能相信。

### 8.4 `HAL_FLASH_Program()`

```c
HAL_FLASH_Program(FLASH_TYPEPROGRAM_HALFWORD,
                  FLASH_LAST_PAGE,
                  0x1234U);
```

指定半字编程，将 `0x1234` 写入目标地址。

这个 API 也会返回 `HAL_StatusTypeDef`。真实工程必须检查返回值，再读回 Flash 做二次确认。

### 8.5 `HAL_FLASH_Lock()`

重新锁定 Flash 控制器，对应设置 `FLASH->CR.LOCK`。

### 8.6 HAL 版读回

HAL 版仍直接用指针读取 Flash 地址。读 Flash 不需要 HAL API，因为它就是存储器映射地址。

### 8.7 HAL 版风险边界

当前 HAL 版没有检查擦除和编程返回值。真实工程应检查返回值和 `err`，再决定是否认为写入成功。

## 9. 两个版本真正应该怎么学

寄存器版重点看：

```text
LOCK/KEYR -> BSY -> PER/AR/STRT -> PG -> 半字写 -> LOCK
```

HAL 版重点看：

```text
Unlock -> EraseInit -> Erase -> Program -> Lock
```

Flash 写入最容易被忽略的是“页擦除会影响整页”，不要只盯着一个半字。

## 10. 检验问题清单

### 10.1 内部 Flash 能像 SRAM 一样直接写吗？

**答**：不能。读可以直接按地址读，写必须解锁、擦除、编程，并等待 Flash 控制器完成。

### 10.2 为什么写前要擦除？

**答**：Flash 单元编程能力受擦除态限制，通常要先把整页擦到擦除态，再按半字写入。

### 10.3 本课为什么选择 `0x0800FC00`？

**答**：它是 64KB Flash 最后一页起始地址，通常避开前面程序代码区域。

### 10.4 页擦除会擦掉多少？

**答**：擦除整个页。本课目标页大小通常 1KB，不只是第一个半字。

### 10.5 解锁钥匙写错会怎样？

**答**：Flash 保持锁定，擦除和编程不会按预期执行。

### 10.6 `BSY` 不等会有什么风险？

**答**：可能在 Flash 忙时发起下一步操作，导致操作失败或状态异常。

### 10.7 HAL 版 `HAL_FLASHEx_Erase()` 对应什么？

**答**：对应寄存器版设置页擦除、写地址、启动擦除并等待完成的流程。

### 10.8 真实参数存储还需要考虑什么？

**答**：需要考虑页内其他数据备份、擦写寿命、掉电保护、校验和链接脚本预留区域。

## 11. 工程实现步骤

### 11.1 需求分析

本课要演示最小内部 Flash 写入：一个固定地址，一个半字，一个读回判断。

### 11.2 硬件核查

无需外部硬件。确认工程不会占用最后一页；若程序变大，应检查链接 map，避免擦掉代码。

### 11.3 寄存器路线

读目标地址，不匹配时解锁 Flash，擦除目标页，开启编程，写半字，等待完成并上锁。

### 11.4 HAL 路线

调用 `HAL_FLASH_Unlock()`，填擦除结构体，调用 `HAL_FLASHEx_Erase()`，再 `HAL_FLASH_Program()`，最后 `HAL_FLASH_Lock()`。

### 11.5 工程思维

内部 Flash 可用作少量参数存储，但不能频繁写。写之前必须规划存储页和数据结构，避免擦除时破坏其他内容。

### 11.6 常见工程陷阱

地址选到代码区、忘记页擦除、没等待 BSY、没重新上锁、忽略擦写寿命、掉电中断写入，都会让 Flash 存储不可靠。

## 12. 运行现象

首次运行时若最后一页未保存 `0x1234`，程序会擦除并写入。写入成功后 PC13 快闪。后续复位运行时，读到 `0x1234`，仍快闪。

如果读回不是 `0x1234`，LED 慢闪。

## 13. 常见问题排查

### 13.1 LED 慢闪

说明读回值不是 `0x1234`。检查 Flash 是否解锁、页擦除是否完成、编程是否执行。

### 13.2 程序写完后异常

可能擦除了程序代码或重要数据。检查 `FLASH_LAST_PAGE` 是否真的未被链接脚本使用。

### 13.3 编程不生效

确认先擦除页，再设置 `PG`，按半字写入，且目标地址半字对齐。

### 13.4 HAL 版无错误但没写入

检查 `HAL_FLASHEx_Erase()` 和 `HAL_FLASH_Program()` 返回值，不要忽略错误码。

### 13.5 多次写入后数据异常

Flash 有擦写寿命限制。频繁写同一页需要磨损均衡或减少写入次数。

## 14. 本课最核心的结论

1. 内部 Flash 可直接读，但不能像 RAM 一样直接随意写。
2. 写 Flash 必须先解锁，再擦除，再编程，最后上锁。
3. 擦除单位是页，写入单位本课使用半字。
4. `BSY` 是判断 Flash 控制器是否忙的关键标志。
5. 最后一页地址必须结合芯片容量和页大小计算。
6. HAL Flash API 封装流程，但不会替你规划安全存储区域。
