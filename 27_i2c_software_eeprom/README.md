# 第 27 课：软件 I2C 与 EEPROM

## 1. 本课到底在学什么

本课表面现象是：不用 STM32 的 I2C1 外设，而是用 PB6/PB7 两个普通 GPIO 手工模拟 I2C 时序，向 AT24C02 发送一段写入序列，并让 PC13 LED 周期翻转。

当前代码演示的是最小软件 I2C 写序列：

```text
START -> 0xA0 -> 0x00 -> 0x5A -> STOP
```

真正要学的是：I2C 不一定只能由 I2C 外设产生。只要软件按 I2C 规则控制 SCL/SDA 的高低电平，也能“造”出从机能看懂的总线波形。

这节课的边界也要说清楚：当前代码只演示发送字节，没有读取 ACK，也没有读回 EEPROM 校验。所以 PC13 翻转只能说明程序在运行、波形在持续输出，不能单独证明 EEPROM 一定写成功。要确认写入是否成功，应加 ACK 读取、读回校验，或使用逻辑分析仪观察总线。

## 2. 本课学习目标

学完本课，你应该能回答：

1. 软件 I2C 和硬件 I2C 的区别是什么？
2. 为什么软件 I2C 仍然必须遵守开漏和上拉规则？
3. `SDA_HIGH()` 为什么不是“强推高”，而是释放 SDA？
4. START 和 STOP 在 SCL/SDA 上分别是什么电平变化？
5. 为什么写一个字节要循环 8 次？
6. 为什么每一位要先放 SDA，再拉高 SCL？
7. 当前代码为什么没有真正判断 AT24C02 是否 ACK？
8. HAL 版的 `HAL_GPIO_WritePin()` 对应寄存器版的哪种 GPIO 写法？

## 4. 实验硬件

- 开发板：STM32F103C8T6 BluePill
- 下载器：ST-Link
- 外部模块：AT24C02 EEPROM
- 软件 I2C 引脚：PB6 = SCL，PB7 = SDA
- LED：PC13，常见 BluePill 为低电平点亮

接线：

```text
AT24C02 VCC -> 3.3V
AT24C02 GND -> GND
AT24C02 SCL -> PB6
AT24C02 SDA -> PB7
```

SCL/SDA 需要上拉电阻。HAL 版启用了内部上拉，但工程上更推荐外部 4.7k 左右上拉，因为 I2C 的上升沿质量直接影响通信可靠性。

## 5. 先建立一个最基本的脑图

```text
GPIOB 时钟打开
  -> PB6/PB7 配成开漏输出
  -> 写 1 表示释放总线，由上拉拉高
  -> 写 0 表示主动拉低总线

i2c_start()
  -> SCL 高时 SDA 从高到低

i2c_write_byte()
  -> 从最高位开始
  -> SDA 放当前 bit
  -> SCL 拉高，让从机采样
  -> SCL 拉低，准备下一位
  -> 第 9 个时钟释放 SDA，理论上从机可 ACK

i2c_stop()
  -> SCL 高时 SDA 从低到高

main()
  -> 反复发送 0xA0、0x00、0x5A
  -> 翻转 PC13 表示循环仍在运行
```

这节课要把 I2C 外设隐藏起来，直接观察“协议时序本身”。这对理解硬件 I2C 的 START、STOP、ACK、TXE、BTF 很有帮助。

## 6. 先认识本课里出现的核心名词

### 6.1 `软件 I2C` 是什么

软件 I2C 是用普通 GPIO 手动模拟 I2C 的 SCL/SDA 时序。

它属于 C 软件层和 GPIO 寄存器层之间的实现方式，不依赖 I2C1 的 `CR1/CCR/SR1` 等寄存器。CPU 每执行一步 GPIO 写操作，总线上就产生相应电平。

它控制的行为是总线波形本身。优点是引脚灵活、容易理解协议；缺点是占用 CPU、速度不精确、不如硬件 I2C 自动处理 ACK 和状态。

### 6.2 `GPIOB` 是什么

`GPIOB` 是 GPIOB 外设寄存器组。

本课 PB6/PB7 不作为 I2C1 复用功能，而是作为普通开漏 GPIO 输出。寄存器版通过 `GPIOB->CRL` 配模式，通过 `GPIOB->BSRR/BRR` 拉高或拉低。

如果 GPIOB 时钟没开，PB6/PB7 配置不会生效，总线没有正确波形。

### 6.3 `PB6 / PB7` 是什么

PB6/PB7 在上一课是 I2C1 默认引脚；本课仍用这两个脚，但不交给 I2C1 外设，而是由软件直接控制。

- PB6：软件 SCL
- PB7：软件 SDA

这说明引脚功能取决于 GPIO 模式和外设复用配置。同一个引脚既可以给 I2C1，也可以当普通 GPIO 使用。

本课选择 PB6/PB7 还有一个教学原因：你可以直接把软件 I2C 波形和硬件 I2C1 默认引脚联系起来。区别不在外部接线，而在芯片内部是谁控制这两根线。硬件 I2C 时，I2C1 外设根据状态机自动翻转 SCL/SDA；软件 I2C 时，CPU 执行宏或 HAL GPIO API 手动翻转。

### 6.4 `开漏输出` 是什么

开漏输出只能主动拉低，不能主动推高。写高电平时实际是释放引脚，让上拉电阻把电平拉高。

软件 I2C 必须使用开漏，因为 SDA 第 9 个时钟要让从机有机会拉低 ACK。如果主机用推挽强推高，从机拉低时就会电气冲突。

### 6.5 `SDA_HIGH()` 是什么

寄存器版：

```c
#define SDA_HIGH() (GPIOB->BSRR = GPIO_BSRR_BS7)
```

在开漏模式下，它表示释放 SDA，不是强行输出高电平。

它属于 C 宏层，对应 GPIO 寄存器写操作。若 PB7 不是开漏，这个宏的硬件意义就会变成推高 SDA，I2C 电气特性就错了。

### 6.6 `SDA_LOW()` 是什么

```c
#define SDA_LOW() (GPIOB->BRR = GPIO_BRR_BR7)
```

它表示主动把 SDA 拉低。START 的关键就是 SCL 高时执行 `SDA_LOW()`。

如果 SDA 无法被拉低，常见原因是引脚没配置好、线接错，或外部模块异常。

### 6.7 `SCL_HIGH()` / `SCL_LOW()` 是什么

这两个宏控制 PB6，也就是软件 I2C 的时钟线。

I2C 数据通常在 SCL 高电平期间被从机采样，所以软件写一位时必须先设置 SDA，再拉高 SCL，延时后再拉低 SCL。

如果 SCL 太快，AT24C02 可能采样不稳定；如果延时太长，通信变慢但通常仍可工作。

### 6.8 `START` 是什么

START 是 SCL 为高时 SDA 从高变低。

软件版代码：

```c
SDA_HIGH();
SCL_HIGH();
soft_i2c_delay();
SDA_LOW();
soft_i2c_delay();
SCL_LOW();
```

这段代码完全是在 GPIO 层手工制造硬件 I2C 外设自动产生的起始条件。

### 6.9 `STOP` 是什么

STOP 是 SCL 为高时 SDA 从低变高。

软件版代码先保证 SDA 低，再拉高 SCL，最后释放 SDA。这个波形告诉从机本次传输结束。

### 6.10 `ACK` 是什么

ACK 是第 9 个时钟期间接收方拉低 SDA。

当前代码在发送 8 位后执行：

```c
SDA_HIGH();
SCL_HIGH();
soft_i2c_delay();
SCL_LOW();
```

这只是释放 SDA 并给出第 9 个时钟，但没有读取 SDA 判断从机是否真的拉低。因此本课代码没有 ACK 校验。

ACK 属于 I2C 协议层，也是判断外部器件是否真正响应的第一道证据。主机发送地址或数据后，接收方如果听懂并接受，就在第 9 个 SCL 高电平期间把 SDA 拉低。因为 SDA 是开漏线，主机必须先释放 SDA，从机才有机会把它拉低。

当前代码没有把 PB7 临时当输入读 `IDR`，所以它只是在“给 ACK 留位置”，没有“检查 ACK 结果”。这也是本课现象必须谨慎解释的原因：LED 翻转说明代码继续运行，逻辑分析仪能看到波形说明主机发出了时序，但 EEPROM 是否接收成功还需要 ACK 或读回证明。

### 6.11 `delay_cycles()` 是什么

软件 I2C 的速度由延时决定。`delay_cycles()` 内部循环执行 `__NOP()`，让 SCL/SDA 的电平保持足够时间。

它属于 C 软件时序层。系统时钟变了，实际 I2C 速度也会跟着变，因为延时循环不是硬件定时器。

### 6.12 `HAL_GPIO_WritePin()` 是什么

HAL 版用它设置 PB6/PB7 输出状态。

在开漏输出模式下，写 `GPIO_PIN_SET` 表示释放总线，写 `GPIO_PIN_RESET` 表示拉低总线。它对应寄存器版的 `BSRR/BRR` 写操作。

### 6.13 `AT24C02 写周期` 是什么

AT24C02 这类 EEPROM 收到写命令后，不是 STOP 一出现就立刻完成内部存储。它还需要一个内部写周期，把数据真正写入非易失存储单元，这个时间通常是毫秒级。

当前代码每秒发送一次写序列，间隔很长，所以通常不会撞上写周期。但如果你改成连续快速写多个地址，就必须在写后延时，或者用 ACK polling 不断发送器件地址，直到 EEPROM 再次 ACK。否则下一次写或读可能发生在器件忙碌期间。

### 6.14 `读回校验` 是什么

读回校验是写 EEPROM 后再读同一地址，确认读出的值是否等于刚才写入的值。

本课代码没有读函数，所以不能做校验。完整 EEPROM demo 至少应包含：写地址和数据、等待写周期、重新发送地址、读回数据、比较结果。没有这一步，LED 翻转和波形存在都只能证明“主机尝试写了”，不能证明“EEPROM 保存了”。

### 6.15 `0xA0` 是什么

`0xA0` 是常见 AT24C02 在 A2/A1/A0 地址脚都为 0 时的写地址字节。它来自 7 位器件地址 `0x50` 左移一位，再加写方向位 0。

如果 EEPROM 模块的地址脚被接成其他状态，7 位地址会变化，写地址字节也会变化。地址不匹配时，从机不会 ACK，当前代码又没有读 ACK，所以你只能从逻辑分析仪或补 ACK 函数后看到问题。

## 7. 寄存器版代码逐步讲解

### 7.1 系统时钟

`system_clock_72mhz_init()` 仍把系统配置到 72MHz。软件 I2C 没有 `CCR` 这类硬件分频寄存器，所以时钟变化会直接影响 `delay_cycles()` 的真实时间。

### 7.2 PC13 LED

`pc13_led_init()` 配置 PC13 输出，`pc13_toggle()` 用来证明主循环在持续运行。它不证明 EEPROM 已写成功。

### 7.3 `app_init()` 打开 GPIOB

```c
RCC->APB2ENR |= RCC_APB2ENR_IOPBEN;
```

软件 I2C 只需要 GPIOB 时钟，不需要打开 `RCC_APB1ENR_I2C1EN`。

### 7.4 PB6/PB7 开漏配置

```c
GPIOB->CRL |= GPIO_CRL_MODE6 | GPIO_CRL_CNF6 |
              GPIO_CRL_MODE7 | GPIO_CRL_CNF7;
```

`MODE=11`，`CNF=11`，表示 50MHz 开漏输出。这里不是复用开漏，因为没有交给 I2C1 外设。

### 7.5 释放总线

```c
SDA_HIGH();
SCL_HIGH();
```

初始化后先释放两根线，让总线处于空闲高电平。若总线空闲时不是高电平，说明上拉、电源、接线或引脚模式有问题。

### 7.6 `i2c_start()`

函数按 START 条件改变 SDA/SCL。重点是 SDA 下降沿发生在 SCL 高电平期间，从机才会识别为起始条件。

### 7.7 `i2c_stop()`

函数按 STOP 条件释放 SDA。重点是 SDA 上升沿发生在 SCL 高电平期间，从机才会结束本次事务。

### 7.8 `i2c_write_byte()`

```c
for(uint8_t i=0;i<8;i++){
    if(b&0x80U) SDA_HIGH(); else SDA_LOW();
    SCL_HIGH();
    soft_i2c_delay();
    SCL_LOW();
    b <<= 1;
}
```

每次发送最高位，发送后左移。I2C 规定高位先发，所以 `0xA0` 会从 bit7 到 bit0 依次放到 SDA。

### 7.9 第 9 个时钟

发送 8 位后，代码释放 SDA 并拉高 SCL。这是 ACK 位的位置。

当前版本没有读 `GPIOB->IDR`，所以不会知道从机是否 ACK。要做完整软件 I2C，应在 SCL 高时读取 SDA。

完整 ACK 读取通常要做三件事：先释放 SDA，让从机能驱动总线；再拉高 SCL，让 ACK 位进入有效采样窗口；最后读取 SDA 电平，低电平表示 ACK，高电平表示 NACK。读完后再拉低 SCL，结束第 9 个时钟。

由于 F103 的开漏输出写 1 就是释放总线，所以不一定非要把 PB7 改成输入模式才能读取 ACK；也可以保持开漏输出并写 1，然后读 `GPIOB->IDR & GPIO_IDR_IDR7`。关键是主机绝不能在 ACK 位强推高 SDA。

### 7.10 `main()` 写序列

```c
i2c_start();
i2c_write_byte(0xA0U);
i2c_write_byte(0x00U);
i2c_write_byte(0x5AU);
i2c_stop();
```

这表示向 AT24C02 地址 `0x00` 写入 `0x5A`。是否真的写入，要靠 ACK/读回或工具验证。

## 8. HAL 版代码逐步讲解

### 8.1 `HAL_Init()` 和 SysTick

HAL 版调用 `HAL_Init()`，并提供 `SysTick_Handler()` 调用 `HAL_IncTick()`。主循环使用 `HAL_Delay(1000)` 控制 LED 翻转间隔。

### 8.2 `soft_i2c_init()`

```c
gpio.Pin = SCL_PIN | SDA_PIN;
gpio.Mode = GPIO_MODE_OUTPUT_OD;
gpio.Pull = GPIO_PULLUP;
gpio.Speed = GPIO_SPEED_FREQ_HIGH;
HAL_GPIO_Init(GPIOB, &gpio);
```

`GPIO_MODE_OUTPUT_OD` 对应开漏输出。`GPIO_PULLUP` 启用内部上拉，但实际 I2C 建议使用外部上拉。

### 8.3 `bus()` 是什么

```c
HAL_GPIO_WritePin(GPIOB,SCL_PIN,scl);
HAL_GPIO_WritePin(GPIOB,SDA_PIN,sda);
i2c_delay();
```

`bus()` 把“设置 SCL/SDA 并延时”封装成一步。它对应寄存器版的宏写 `BSRR/BRR` 加 `soft_i2c_delay()`。

### 8.4 HAL 版 START/STOP

`i2c_start()` 和 `i2c_stop()` 仍然按 I2C 波形规则操作，只是底层写 GPIO 的方式换成 HAL API。

HAL 没有在这里使用 `HAL_I2C_*`，因为本课故意不用硬件 I2C 外设。

### 8.5 HAL 版写字节

HAL 版 `i2c_write_byte()` 同样高位先发，每一位经历 SCL 低设置 SDA、SCL 高采样、SCL 低结束。

### 8.6 `HAL_GPIO_WritePin()` 与开漏语义

在开漏模式下，`GPIO_PIN_SET` 是释放引脚，`GPIO_PIN_RESET` 是拉低引脚。理解这一点比记 API 名字更重要。

### 8.7 当前 HAL 版同样没有 ACK 判断

HAL 版第 9 个时钟释放 SDA，但没有调用 `HAL_GPIO_ReadPin()` 读取 ACK。它和寄存器版边界一致。

如果要补 HAL 版 ACK，思路和寄存器版一样：先 `HAL_GPIO_WritePin(GPIOB, SDA_PIN, GPIO_PIN_SET)` 释放 SDA，再把 SCL 拉高，然后用 `HAL_GPIO_ReadPin(GPIOB, SDA_PIN)` 读取。读到 `GPIO_PIN_RESET` 才表示从机 ACK。

这也说明 HAL GPIO 只是换了一种写法，并不会自动把软件 I2C 变成完整协议栈。START、STOP、数据位、ACK、超时、总线恢复都仍然由你自己设计。

### 8.9 HAL 内部上拉和外部上拉

HAL 版配置了 `gpio.Pull = GPIO_PULLUP`，这能让 PB6/PB7 在释放时回到高电平。但 STM32 内部上拉阻值通常较大，上升沿会比较慢，线长稍长或总线电容稍大时，I2C 波形可能变钝。

工程上更推荐在 SCL/SDA 上接 4.7k 左右外部上拉。内部上拉适合入门验证，外部上拉适合可靠通信。看到逻辑分析仪上高电平上升很慢、ACK 偶尔失败、速度稍高就不稳定时，优先检查上拉。

### 8.10 软件 I2C 的超时意识

本课代码为了突出波形，没有做超时。真实软件 I2C 不应该无限等待或盲目发送，因为总线可能被从机拉低、接线可能断开、ACK 可能失败。

完整驱动里，START 前应检查总线是否空闲，写字节后应检查 ACK，必要时产生若干 SCL 脉冲尝试释放被卡住的 SDA。否则一次异常就可能让后续通信都看似“没反应”。

### 8.8 `HAL_Delay(1000)`

主循环每秒发送一次写序列并翻转 LED。LED 翻转说明代码没卡住，不能直接说明 EEPROM 写成功。

## 9. 两个版本真正应该怎么学

寄存器版看 GPIO 寄存器怎样直接改变引脚电平；HAL 版看同样的引脚动作如何被 API 封装。

两者都没有使用 I2C1 外设，所以不要拿 `I2C1->SR1`、`HAL_I2C_Mem_Write()` 来解释本课代码。本课重点是“协议波形可以由软件手工产生”。

## 10. 检验问题清单

### 10.1 软件 I2C 为什么能工作？

**答**：因为 I2C 从机识别的是 SCL/SDA 上符合协议的电平时序。只要 GPIO 按规则产生 START、地址、数据、ACK 时钟和 STOP，从机就能识别。

### 10.2 为什么 `SDA_HIGH()` 叫释放 SDA 更准确？

**答**：PB7 是开漏输出，写 1 时并不是强推高，而是关闭下拉管，让上拉电阻把 SDA 拉高。

### 10.3 当前代码有没有判断 ACK？

**答**：没有。代码给出了第 9 个时钟并释放 SDA，但没有读取 SDA 电平判断从机是否拉低。

### 10.4 LED 翻转能证明 EEPROM 写成功吗？

**答**：不能。LED 翻转只能证明主循环在运行。写成功需要 ACK 判断、读回校验或逻辑分析仪确认。

### 10.5 为什么每位要先设置 SDA 再拉高 SCL？

**答**：I2C 从机通常在 SCL 高电平期间采样 SDA。先放好数据再拉高 SCL，才能让从机读到稳定电平。

### 10.6 软件 I2C 的速度由什么决定？

**答**：由 CPU 执行速度和延时循环决定。系统时钟或编译优化变化都会影响实际 SCL 频率。

### 10.7 HAL 版使用的是硬件 I2C 吗？

**答**：不是。HAL 版只用 HAL GPIO API 控制 PB6/PB7，并没有调用 `HAL_I2C_*`。

### 10.8 如果要补 ACK，应该怎么做？

**答**：发送 8 位后释放 SDA，在 SCL 高电平期间读取 SDA；低电平表示 ACK，高电平表示 NACK。

## 11. 工程实现步骤

### 11.1 需求分析

本课目标是理解 I2C 波形本身，不是做完整 EEPROM 驱动。因此代码保持最小：只产生写序列并用 LED 表示循环节奏。

### 11.2 硬件核查

确认 PB6/PB7 连接 AT24C02 SCL/SDA，模块供电 3.3V，GND 共地，SCL/SDA 有上拉。

### 11.3 寄存器路线

打开 GPIOB，配置 PB6/PB7 开漏输出，用 `BSRR/BRR` 实现拉低和释放，再手写 START、STOP、写字节。

### 11.4 HAL 路线

用 `GPIO_InitTypeDef` 配 PB6/PB7 为 `GPIO_MODE_OUTPUT_OD`，用 `HAL_GPIO_WritePin()` 按同样时序控制两根线。

### 11.5 工程思维

软件 I2C 适合救急、调试或引脚不够灵活时使用。正式项目要考虑 CPU 占用、时序精度、ACK/超时、总线恢复和多设备冲突。

它的最大优点是透明：每一次 SCL/SDA 变化都在你的代码里，适合用来学习协议和排查硬件。它的最大缺点也是透明：所有时序都靠 CPU 执行，主循环一忙、关中断时间一长、编译优化一变，I2C 速度都可能变化。

所以软件 I2C 不应只追求“能发波形”。工程版本至少要有 ACK 判断、超时、读回校验和错误返回值。只有这样，上层代码才能知道 EEPROM 是真的写成功，还是只是 MCU 自己把波形发出去了。

### 11.6 常见工程陷阱

不要把“释放高电平”理解成“推挽输出高”。不要用 LED 翻转替代通信校验。不要忘记 ACK 需要读取 SDA，不是只产生第 9 个时钟。

## 12. 运行现象

下载后 PC13 会按约 1 秒间隔翻转。若接了逻辑分析仪，PB6/PB7 上能看到周期性 I2C 写序列：起始、`0xA0`、`0x00`、`0x5A`、停止。

如果 EEPROM 接线正确且时序足够，它可能被写入 `0x5A`；但当前代码没有读回验证。

## 13. 常见问题排查

### 13.1 PC13 不翻转

先查程序是否下载成功、PC13 是否为低电平亮、GPIOC 时钟和模式是否正常。

### 13.2 逻辑分析仪看不到 SCL/SDA

检查 PB6/PB7 是否接错、GPIOB 时钟是否打开、探头地线是否共地。

### 13.3 SCL/SDA 高电平上不去

检查上拉电阻、模块供电、是否有器件长期拉低总线。开漏输出释放后必须能回到高电平。

### 13.4 EEPROM 没写进去

当前代码不判断 ACK，也不等待写周期和读回。先补 ACK 读取，再加写后延时和读回校验。

如果 ACK 正常但读回不对，再检查 AT24C02 地址脚 A0/A1/A2 是否和 `0xA0` 匹配。`0xA0` 对应常见 7 位基地址 `0x50` 左移后的写地址，地址脚状态改变后，器件地址也会改变。

还要注意 EEPROM 页写规则。AT24C02 通常有页写限制，连续写跨页时地址会在页内回卷。本课只写一个字节，不涉及页回卷；后续扩展多字节写时必须单独处理。

### 13.5 波形太快或太慢

调整 `soft_i2c_delay()` 或 `i2c_delay()`。软件延时受系统时钟和编译优化影响，不能当精确定时器。

## 14. 本课最核心的结论

1. 软件 I2C 的本质是用 GPIO 手工产生 I2C 总线波形。
2. 开漏和上拉仍然是 I2C 的电气基础，软件模拟也不能跳过。
3. START、STOP、数据位和 ACK 都是 SCL/SDA 的具体电平关系。
4. 当前代码只演示发送写序列，没有验证 ACK 和读回结果。
5. HAL GPIO 版不是硬件 I2C，只是把 GPIO 写寄存器换成 API。
6. 理解软件 I2C 后，硬件 I2C 的状态标志会更容易理解。

## 15. 建议你现在怎么读这节课

先用纸画出 START、发送 `0xA0`、ACK 位和 STOP 的 SCL/SDA 波形。再对照 `i2c_write_byte()` 看每一次宏调用对应波形上的哪一段。最后用逻辑分析仪验证代码产生的波形。
